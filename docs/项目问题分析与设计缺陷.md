# 项目问题分析与设计缺陷

> 更新日期：2026-04-15  
> 基于当前代码现状的系统性分析

## 一、关键缺陷（建议优先修复）

### 1.1 子站中间件 `_nav_token` 清洗逻辑不可达

**位置**：`subsite-middleware/auth_check.php` 第 37-49 行

**问题描述**：
`_nav_token` 的 Cookie 写入和 URL 清洗逻辑位于未登录分支的 `exit` 之后：

```php
if (!$user) {
    header('Location: ' . auth_nav_login_url() . '?redirect=' . $redirect);
    exit;  // ← 这里已经 exit 了
}

// 下面的代码在常规执行流中永远不会执行
if (!empty($_GET['_nav_token']) && empty($_COOKIE[SESSION_COOKIE_NAME])) {
    auth_set_cookie($_GET['_nav_token']);
    // ... 清洗 URL 并 302
}
```

**影响**：子站通过 URL 参数 `_nav_token` 首次传递 Token 后，无法自动写入 Cookie 并清洗 URL。用户每次访问子站都必须携带 `_nav_token` 参数，否则会被重定向到登录页。

**修复建议**：将 `_nav_token` 处理逻辑前移至 `auth_get_current_user()` 之后、`if (!$user)` 之前：

```php
$user = auth_get_current_user();

// 如果 Token 来自 URL 参数，先写入 Cookie
if (!empty($_GET['_nav_token']) && empty($_COOKIE[SESSION_COOKIE_NAME])) {
    auth_set_cookie($_GET['_nav_token']);
    $clean_url = ...;
    header('Location: ' . $clean_url);
    exit;
}

if (!$user) {
    // 未登录跳转
}
```

### 1.2 `auth_needs_setup()` 逻辑冗余且存在重复判断

**位置**：`shared/auth.php` 第 334-344 行

**问题描述**：

```php
function auth_needs_setup(): bool {
    if (file_exists(INSTALLED_FLAG)) {
        return false;
    }
    if (!empty(auth_load_users_raw())) {
        return false;
    }
    $users = auth_load_users();
    return empty($users);
}
```

`auth_load_users_raw()` 已经读取了磁盘上的用户（不含开发模式虚拟用户），如果它不为空，直接返回 `false`。但接下来又调用 `auth_load_users()`（会注入开发模式虚拟用户），再判断一次是否为空。这意味着：
- 生产环境：`auth_load_users_raw()` 为空时，`auth_load_users()` 也必然为空，第二次判断是多余的。
- 开发环境：`auth_load_users()` 会因虚拟用户注入而永远非空，导致 `auth_needs_setup()` 永远返回 `false`，开发模式下如果 `.installed` 缺失也不会进入安装向导。这实际上符合预期（避免反复安装），但代码逻辑表达了两种不同的意图，造成阅读困惑。

**修复建议**：明确注释开发模式的短路行为，或直接简化：

```php
function auth_needs_setup(): bool {
    if (file_exists(INSTALLED_FLAG)) {
        return false;
    }
    // 开发模式注入虚拟用户，无需重复进入安装向导
    return empty(auth_load_users_raw());
}
```

### 1.3 Webhook HTTP 发送逻辑重复

**位置**：
- `admin/shared/functions.php`：`webhook_http_post_json()`（约 60 行）
- `shared/notify_runtime.php`：`notify_http_post_json()`（约 60 行）

**问题描述**：两个函数几乎完全一致，都是构造 curl / stream context 发送 JSON POST 请求。维护时容易遗漏同步修改（如超时调整、协议限制、重定向策略等）。

**修复建议**：将通用 HTTP POST JSON 能力下沉到 `shared/` 中的独立文件（如 `shared/http_client.php`），两处统一调用。

---

### 1.4 `public/index.php` 职责过重

**位置**：`public/index.php`

**问题描述**：该文件同时承担了：
1. 业务逻辑（用户认证、站点数据过滤、健康状态读取）
2. HTML 结构渲染
3. 内联 CSS 生成（根据 `card_layout`/`card_direction` 动态输出样式规则）
4. 内联 JavaScript（搜索、Tab 切换、最近访问、过滤逻辑）

文件长达 811 行，混合了 PHP 后端逻辑与前端呈现，可读性和可维护性差。

**修复建议**：
- 将动态 CSS 规则抽离到独立 PHP 文件（如 `public/shared/homepage_css.php`）
- 将 JS 抽离到 `public/assets/homepage.js`
- 将业务逻辑函数（`homepage_group_is_visible()`, `homepage_render_site_card()` 等）抽离到 `public/shared/homepage_lib.php`

## 二、中等风险问题

### 2.1 `debug_read_log()` 大文件读取性能隐患

**位置**：`admin/shared/functions.php` 第 450-495 行

**问题描述**：使用 PHP 原生 `fseek/fread` 从文件末尾向前 chunk 读取以获取最后 N 行。虽然比一次性读入内存好，但在超大日志文件（GB 级别）时，仍然需要多次磁盘寻道和读取，性能不理想。且 `chunk = 8192` 固定，未根据文件大小自适应。

**修复建议**：
- 增加文件大小上限判断，超过阈值直接拒绝在线查看或仅提示下载
- 优先使用 `exec('tail -n ' . $lines . ' ' . escapeshellarg($path))`（在允许 shell 执行的环境中更快）
- 或增加分页参数，避免一次性请求大量行数

---

### 2.2 `host_agent_lib.php` 中 `health_build_curl_handle` 未被使用

**位置**：`admin/shared/host_agent_lib.php` 第 581-595 行

**问题描述**：定义了 `health_build_curl_handle()` 函数，但在同一文件中没有任何调用点。该函数似乎是遗留代码或健康检查逻辑迁移时的残留。

**修复建议**：确认是否已在其他文件中被引用（全局搜索），如无引用则删除。

---

### 2.3 `admin_run_command()` 使用 `exec` 且无超时控制

**位置**：`admin/shared/functions.php` 第 108-117 行

**问题描述**：

```php
function admin_run_command(string $command): array {
    exec($command . ' 2>&1', $output, $code);
    return ['ok' => $code === 0, 'code' => $code, 'output' => implode("\n", $output)];
}
```

`exec()` 没有超时参数，如果执行的命令挂起（如 `nginx -s reload` 在某些锁竞争场景），会导致 PHP-FPM 进程长期阻塞，最终超时返回 502。

**修复建议**：使用 `proc_open()` + `stream_select()` 实现带超时的命令执行，或限制为仅调用已知快速的命令。

---

### 2.4 `backup_restore()` 未验证备份文件内容结构

**位置**：`admin/shared/functions.php` 第 322-336 行

**问题描述**：恢复备份时仅校验文件名格式和 `json_decode()` 是否成功，未校验解析后的数据结构是否包含必需的键（如 `sites` 是否为数组）。如果用户上传了一个格式合法但结构错误的 JSON，可能导致后续 `backup_apply_restored_sections()` 写入非法数据。

**修复建议**：增加结构校验：

```php
if (!isset($data['sites']) || !is_array($data['sites'])) {
    return false;
}
```

---

### 2.5 `dns_api_upsert_resolved()` 缓存更新与并发安全

**位置**：`admin/shared/dns_api_lib.php`

**问题描述**：`zoneRecordsCache` 是一个按引用传递的内存数组，用于减少同一批次中对同一 Zone 的重复 API 调用。但在并发场景下（多个请求同时调用 DNS API），该缓存无法生效，且 `dns_api_upsert_resolved()` 内部没有文件锁或乐观锁机制，两个并发请求同时更新同一条记录时可能导致后写覆盖。

**修复建议**：
- 对关键 DNS 配置文件更新增加 `LOCK_EX` 文件锁
- 或在更新前增加二次确认（读取最新记录状态后再决定 update/create）

---

### 2.6 `get_client_ip()` 对 X-Forwarded-For 链的信任度过高

**位置**：`shared/auth.php` 第 970-1000 行

**问题描述**：`get_client_ip()` 优先读取 `HTTP_X_FORWARDED_FOR` 链中最左侧的公网 IP。如果 Nginx 未正确配置仅传递可信上游设置的 X-Forwarded-For，客户端可以伪造该头部，导致 IP 锁定绕过或日志污染。

**修复建议**：
- 增加可信代理 IP 列表配置，仅接受来自可信代理的 X-Forwarded-For
- 或增加环境变量 `TRUSTED_PROXIES`，在函数内进行来源校验

## 三、设计层面的改进建议

### 3.1 缺少统一的异常处理机制

**现状**：大量 PHP 文件在出现错误时直接调用 `die()`、`exit`、或返回硬编码的 HTTP 状态码。没有统一的异常捕获层，也没有标准化的错误日志记录规范。

**建议**：
- 引入轻量级异常基类（如 `NavException`）
- 在 API 入口文件顶部使用 `try/catch` 统一包裹，将异常转换为标准 JSON 错误响应
- 区分 "用户可感知错误" 和 "系统内部错误"

### 3.2 配置读取存在多处重复实现

**现状**：
- `shared/auth.php`：`auth_get_config()`
- `admin/shared/functions.php`：`load_config()`
- `admin/shared/dns_lib.php`：`load_dns_config()`
- 各处还有其他小型的 JSON 文件读取函数

这些函数模式高度相似（检查文件存在 → `json_decode` → 合并默认值），但没有统一的抽象。

**建议**：创建一个通用的 `json_config_read($path, $defaults = [])` 函数，下沉到 `shared/` 中供全局使用。

### 3.3 权限模型未完全细化到功能点

**现状**：角色权限映射表在 `shared/auth.php` 中定义，但后台页面中大量地方仍然直接使用 `auth_require_admin()`，而不是按功能点检查 `auth_user_has_permission()`。例如文件管理器理论上需要 `fs.write`、`fs.read` 等细粒度权限，但当前实现中这些权限点尚未在系统中注册。

**建议**：
- 完成权限点枚举（如 `fs.view`, `fs.write`, `fs.delete`, `ssh.manage`, `host.runtime.view` 等）
- 逐步将页面中的 `auth_require_admin()` 替换为 `auth_require_permission()`
- 在角色管理界面支持自定义权限勾选

### 3.4 测试分层结构不完整

**现状**：
- 页面级 Full E2E 已经较完整（90 个 spec）
- CLI 回归已经补充主要契约
- 但 `shared/`、`admin/shared/`、`subsite-middleware/` 缺少底层单元/集成测试
- 移动端测试未纳入默认回归基线

**建议**：
- 建立 "页面级 E2E + 底层集成测试 + CLI 回归" 的三层测试结构
- 使用 `runDockerPhpInline` 对 `shared/auth.php`、`shared/notify_runtime.php` 等库函数编写底层集成测试
- 将 `mobile-chrome` 纳入 CI 默认执行矩阵

### 3.5 前端代码组织松散

**现状**：大量内联 CSS/JS 分散在各个 PHP 页面中，没有模块化和构建流程。随着功能增加，页面体积和重复代码会继续膨胀。

**建议**（长期）：
- 引入前端构建工具（如 Vite）将 JS/CSS 抽离为独立模块
- 使用现代前端框架（如 Vue/React/Svelte）重构交互密集页面（如文件管理器、计划任务、DDNS）
- 短期内可先统一将内联脚本抽离到 `public/assets/` 和 `admin/assets/`

## 四、已发现但当前可接受的妥协

| 问题 | 说明 | 接受原因 |
|------|------|----------|
| JSON 文件存储无事务性 | 并发写可能导致文件损坏 | 项目定位轻量，单用户/低并发场景风险可控；已使用 `LOCK_EX` 缓解 |
| 没有数据库迁移机制 | 新增字段靠代码默认值补全 | 无数据库依赖是项目核心卖点；`auth_default_config()` 等函数已承担类似职责 |
| 健康检查使用同步 HTTP HEAD | 大量站点时页面可能变慢 | 已引入 5 分钟缓存和 curl_multi 并行检测 |
| host-agent 使用自研 TCP HTTP Server | 非标准 Web 服务器 | 为了减少依赖、保持单容器可运行；功能足够简单 |
