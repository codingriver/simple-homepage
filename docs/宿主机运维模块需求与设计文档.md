# 宿主机运维模块需求与设计文档

## 1. 目标

宿主机运维模块用于补齐当前项目在"轻量 1Panel / 宝塔"方向上的主机能力，让后台不仅能管理导航、计划任务、SSH、文件系统，还能直接管理宿主机运行状态。

本阶段已落地以下能力：

- 系统概览
- 进程管理
- 服务管理
- 网络分析
- 宿主机用户管理
- 宿主机用户组管理

Docker 宿主管理作为独立模块已单独实现，详见 [Docker宿主管理模块需求与设计文档.md](./Docker宿主管理模块需求与设计文档.md)。

## 2. 架构原则

宿主机运维能力继续复用现有 `host-agent` 体系，不在 Web 页面中直接执行系统命令。

统一调用链：

```text
admin/host_runtime.php
  -> admin/host_api.php
  -> admin/shared/host_agent_lib.php
  -> host-agent HTTP API
  -> cli/host_agent.php
```

这样做的原因：

- 本机与远程 / 模拟模式统一
- 权限判断统一
- 审计统一
- 页面层和执行层解耦
- 后续 Docker 管理也能继续复用这一层

## 3. 模式说明

### 3.1 host 模式

- `host-agent` 通过特权容器运行
- 可进入宿主机 PID / mount / network 命名空间
- 真正执行 `systemctl`、`service`、`useradd`、`groupadd`、`ss`、`ps` 等命令

### 3.2 simulate 模式

- 不改真实宿主机
- 系统概览、进程、网络可读取 `host-agent` 自身容器环境
- 用户 / 用户组写操作落在 `data/host-agent-sim-root/var/lib/host-agent/` 里的模拟状态文件
- 服务管理只保留 SSH 的模拟动作

## 4. 功能拆分

### 4.1 系统概览

展示：

- 主机名
- 操作系统
- 内核版本
- CPU 型号 / 核心数
- CPU 使用率
- 1 / 5 / 15 分钟负载
- 内存 / Swap
- 根分区磁盘使用情况

对应 API：

- `GET /system/overview`

### 4.2 进程管理

支持：

- 按 CPU / 内存排序
- 关键字搜索
- 查看 PID、用户、CPU、内存、状态、命令
- 结束进程
- 强制结束进程

对应 API：

- `POST /process/list`
- `POST /process/kill`

### 4.3 服务管理

支持：

- 列出服务
- 查看 active / sub / enabled
- 启动 / 停止 / 重启 / 重载
- 查看最近服务日志

对应 API：

- `POST /service/list`
- `POST /service/action`
- `POST /service/logs`

说明：

- 真实 `host` 模式优先面向 systemd
- `simulate` 模式仅保留 SSH 服务的模拟状态

### 4.4 网络分析

支持：

- 查看监听端口
- 查看已建立连接
- 方便分析哪个服务占端口、当前连接是否异常

对应 API：

- `POST /network/overview`

### 4.5 用户管理

支持：

- 查看用户列表
- 新增 / 修改用户
- 设置 shell / home / groups / gecos
- 设置密码
- 锁定 / 解锁用户
- 删除用户

对应 API：

- `POST /user/list`
- `POST /user/save`
- `POST /user/password`
- `POST /user/lock`
- `POST /user/delete`

### 4.6 用户组管理

支持：

- 查看用户组列表
- 新增用户组
- 设置组成员
- 删除用户组

对应 API：

- `POST /group/list`
- `POST /group/save`
- `POST /group/delete`

## 5. 权限策略

本阶段复用现有 SSH 权限：

- 读操作：`ssh.view`
- 写操作：`ssh.manage` 或 `ssh.service.manage`

原因：

- 当前项目里主机侧能力已经统一挂在 SSH / Host-Agent 体系下
- 先复用现有角色，能更快落地
- 后续如果宿主机运维模块继续扩大，再拆更细的 `host.runtime.*` 权限

## 6. 审计策略

所有写操作都进入现有 SSH 审计体系，动作包括：

- `host_process_kill`
- `host_service_action`
- `host_user_save`
- `host_user_delete`
- `host_user_password`
- `host_user_lock`
- `host_group_save`
- `host_group_delete`
