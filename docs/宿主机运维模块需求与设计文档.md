# 宿主机运维模块需求与设计文档

## 1. 目标

宿主机运维模块用于补齐当前项目在"轻量 1Panel / 宝塔"方向上的主机能力，让后台不仅能管理导航、计划任务、SSH、文件系统，还能直接管理宿主机运行状态。

本阶段已落地以下能力：

- 系统概览
- 进程管理
- 服务管理
- 网络分析
- 宿主机用户管理
- 宿主机用户组管理

Docker 宿主管理作为独立模块已单独实现，详见 [Docker宿主管理模块需求与设计文档.md](./Docker宿主管理模块需求与设计文档.md)。

## 2. 架构原则

宿主机运维能力继续复用现有 `host-agent` 体系，不在 Web 页面中直接执行系统命令。

统一调用链：

```text
admin/host_runtime.php
  -> admin/host_api.php
  -> admin/shared/host_agent_lib.php
  -> host-agent HTTP API
  -> cli/host_agent.php
```

这样做的原因：

- 本机与远程 / 模拟模式统一
- 权限判断统一
- 审计统一
- 页面层和执行层解耦
- 后续 Docker 管理也能继续复用这一层

## 3. 模式说明

### 3.1 host 模式

- `host-agent` 通过特权容器运行
- 可进入宿主机 PID / mount / network 命名空间
- 真正执行 `systemctl`、`service`、`useradd`、`groupadd`、`ss`、`ps` 等命令

### 3.2 simulate 模式

- 不改真实宿主机
- 系统概览、进程、网络可读取 `host-agent` 自身容器环境
- 用户 / 用户组写操作落在 `data/host-agent-sim-root/var/lib/host-agent/` 里的模拟状态文件
- 服务管理只保留 SSH 的模拟动作

## 4. 功能拆分

### 4.1 系统概览

展示：

- 主机名
- 操作系统
- 内核版本

- 1 / 5 / 15 分钟负载
- 内存 / Swap
- 根分区磁盘使用情况

对应 API：

- `GET /system/overview`

### 4.2 进程管理

支持：

- 按 CPU / 内存排序
- 关键字搜索
- 查看 PID、用户、CPU、内存、状态、命令
- 结束进程
- 强制结束进程

对应 API：

- `POST /process/list`
- `POST /process/kill`

### 4.3 服务管理

支持：

- 列出服务
- 查看 active / sub / enabled
- 启动 / 停止 / 重启 / 重载
- 查看最近服务日志

对应 API：

- `POST /service/list`
- `POST /service/action`
- `POST /service/logs`

说明：

- 真实 `host` 模式优先面向 systemd
- `simulate` 模式仅保留 SSH 服务的模拟状态

### 4.4 网络分析

支持：

- 查看监听端口
- 查看已建立连接
- 方便分析哪个服务占端口、当前连接是否异常

对应 API：

- `POST /network/overview`

### 4.5 用户管理

支持：

- 查看用户列表
- 新增 / 修改用户
- 设置 shell / home / groups / gecos
- 设置密码
- 锁定 / 解锁用户
- 删除用户

对应 API：

- `POST /user/list`
- `POST /user/save`
- `POST /user/password`
- `POST /user/lock`
- `POST /user/delete`

### 4.6 用户组管理

支持：

- 查看用户组列表
- 新增用户组
- 设置组成员
- 删除用户组

对应 API：

- `POST /group/list`
- `POST /group/save`
- `POST /group/delete`

## 5. 权限策略

本阶段复用现有 SSH 权限：

- 读操作：`ssh.view`
- 写操作：`ssh.manage` 或 `ssh.service.manage`

原因：

- 当前项目里主机侧能力已经统一挂在 SSH / Host-Agent 体系下
- 先复用现有角色，能更快落地
- 后续如果宿主机运维模块继续扩大，再拆更细的 `host.runtime.*` 权限

## 6. 审计策略

所有写操作都进入现有 SSH 审计体系，动作包括：

- `host_process_kill`
- `host_service_action`
- `host_user_save`
- `host_user_delete`
- `host_user_password`
- `host_user_lock`
- `host_group_save`
- `host_group_delete`

## 9. 软件包管理（Phase 1）

软件包管理模块通过 host-agent 在宿主机上统一管理各类 Linux/Unix 发行版的软件包。通过抽象层屏蔽不同包管理器的差异，提供统一的安装、卸载、搜索、查询、升级接口。

└─────────────────┘     └──────────────┘     └──────────────────┘

│  │  HTTP Server    │ → │ 包管理器检测     │ → │ 执行命令    │ │

### 9.2 包管理器自动检测

Host-Agent 启动时自动检测宿主机上的包管理器和服务管理器：

**包管理器检测顺序：**

```text
brew → port → apt → dnf → yum → apk → pacman → zypper → emerge → unknown
```

| 包管理器 | 检测命令 | 典型发行版 |
|---------|---------|-----------|
| `brew` | `command -v brew` | macOS (Homebrew) |
| `port` | `command -v port` | macOS (MacPorts) |
| `apt` | `command -v apt-get` | Debian/Ubuntu |
| `dnf` | `command -v dnf` | Fedora/RHEL 8+ |
| `yum` | `command -v yum` | CentOS/RHEL 7 |
| `apk` | `command -v apk` | Alpine Linux |
| `pacman` | `command -v pacman` | Arch Linux |
| `zypper` | `command -v zypper` | openSUSE |
| `emerge` | `command -v emerge` | Gentoo |

**服务管理器检测：**

```text
systemd → openrc → runit → sysvinit → launchd → unknown
```

### 9.3 包名映射机制

不同发行版对同一软件的包名可能不同。Host-Agent 内置 28 个常用软件的跨发行版包名映射。调用 API 时传入通用名（如 `mysql`），自动解析为对应包管理器的实际包名。

| redis | redis-server | redis | redis | redis |
| php-fpm | php-fpm | php-fpm | php82-fpm | php |
| docker | docker.io | docker | docker | docker |

### 9.5 host 模式与 simulate 模式执行差异

| 场景 | host 模式 | simulate 模式 |
|------|----------|--------------|
| 包安装 | `apt-get install -y nginx` 真实安装到宿主机 | 在容器内执行，不修改宿主机 |
| 包卸载 | `apt-get remove -y nginx` 真实卸载 | 在容器内执行模拟命令 |
| 已安装检测 | `dpkg-query -W` 查询宿主机 | 查询容器内的包数据库 |
| 包列表 | 返回宿主机真实已安装列表 | 返回容器内已安装列表 |

**关键代码差异：**

```php
function host_agent_host_shell(string $script, ?string $stdin = null): array {
    $mode = (string)(getenv('HOST_AGENT_MODE') ?: 'host');
    if ($mode === 'simulate') {
        // simulate 模式下不进入宿主机 namespace，直接在容器内执行
        return host_agent_proc_run(['sh', '-lc', $script], $stdin);
    }
    return host_agent_proc_run(
        ['/usr/bin/nsenter', '-t', '1', '-m', '-u', '-i', '-n', '-p', 'sh', '-lc', $script],
        $stdin
    );
}
```

## 10. 配置管理（Phase 2）

配置管理模块用于安全地编辑和管理宿主机上的常用服务配置文件。核心特点是**原子操作**：每次修改前自动备份，校验失败时自动回滚，确保系统配置始终可恢复。

### 10.1 支持的配置项

| 配置项 | 图标 | 配置文件路径 | 格式 | 校验命令 | 重载命令 |
|--------|------|-------------|------|---------|---------|
| `nginx` | 🌐 | `/etc/nginx/nginx.conf` | nginx | `nginx -t` | `nginx -s reload` |
| `php-fpm` | 🐘 | `/etc/php/8.2/fpm/php.ini` | ini | — | — |
| `redis` | 🔴 | `/etc/redis/redis.conf` | redis_conf | — | — |
| `ssh` | 🔐 | `/etc/ssh/sshd_config` | ssh_config | — | — |
| `mysql` | 🐬 | `/etc/mysql/my.cnf` | ini | — | — |
| `postgresql` | 🐘 | `/etc/postgresql/16/main/postgresql.conf` | postgresql_conf | — | — |

> 不同发行版的实际路径由 `host_agent_config_get_path()` 根据检测到的包管理器动态选择。macOS (Homebrew) 路径使用 `/opt/homebrew/etc/` 前缀。

### 10.2 原子操作工作流

配置应用遵循以下原子流程：

```text
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  读取   │ → │  备份   │ → │  写入   │ → │  校验   │ → │  重载   │
│ 当前配置 │    │ 原文件   │    │ 新内容   │    │ validate │    │ reload  │
└─────────┘    └─────────┘    └─────────┘    └────┬────┘    └────┬────┘
                                                   │              │
                                              失败 │         失败 │
                                                   ↓              ↓
                                              ┌─────────┐    ┌─────────┐
                                              │  自动回滚 │    │  自动回滚 │
                                              │ 恢复备份  │    │ 恢复备份  │
                                              └─────────┘    └─────────┘
```

**回滚机制细节：**

1. 每次 `config_apply` 前，原配置文件被复制到 `/tmp/host-agent-config-backup-{configId}-{timestamp}`
2. 若校验命令（如 `nginx -t`）返回非零退出码，自动将备份恢复回原路径
3. 若重载命令返回非零退出码，同样自动回滚
4. 回滚成功后返回错误信息，系统配置保持修改前状态

### 10.3 备份历史与恢复机制

```text
用户请求恢复指定备份
        │
        ▼
┌─────────────────┐
│ 校验备份路径合法性 │  ← 必须是 .backup. 结尾、在同一目录下
│ (防路径遍历攻击)  │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 备份当前配置文件  │  ← 先备份再恢复，确保可二次回退
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 复制备份到原路径  │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 执行重载命令      │  ← 如 nginx -s reload
└─────────────────┘
```

### 10.4 API 接口

| 接口 | 方法 | 权限 | 说明 |
|------|------|------|------|
| `config_definitions` | GET | `ssh.view` | 获取配置定义列表 |
| `config_read` | POST | `ssh.config.manage`/`ssh.manage` | 读取配置内容 |
| `config_apply` | POST | `ssh.config.manage`/`ssh.manage` | 应用配置（含校验+重载） |
| `config_validate` | POST | `ssh.config.manage`/`ssh.manage` | 仅校验不写入 |
| `config_history` | POST | `ssh.config.manage`/`ssh.manage` | 查询备份历史 |

## 11. 声明式管理（Phase 3）

声明式管理允许用户通过 JSON Manifest 描述期望的系统状态，Host-Agent 自动将实际状态对齐到期望状态。支持预演（dry-run）模式，在真正执行前预览所有变更。

### 11.1 Manifest DSL 语法

Manifest 是一个 JSON 对象，包含以下可选顶级字段：

```json
{
  "packages": {
    "nginx": { "state": "installed" },
    "apache": { "state": "absent" }
  },
  "services": {
    "nginx": { "state": "running", "enabled": true },
    "mysql": { "state": "stopped", "enabled": false }
  },
  "configs": {
    "nginx": {
      "sections": {
        "http": { "gzip": "on" }
      }
    }
  },
  "users": {
    "deploy": { "state": "present" }
  }
}
```

### 11.2 三种执行模式

**validate — 校验**
- 校验 JSON 格式和 Schema
- 检查包名映射是否存在
- 不执行任何系统修改

**dry-run — 预演**
- 执行完整的对齐逻辑计算
- 输出所有将要执行的 install/remove/start/stop/enable/disable/config 操作
- 所有变更标记 `dry_run: true`
- 不实际修改系统

**apply — 应用**
- 先执行 schema 校验
- 按顺序执行：packages → services → configs → users
- 每个操作记录执行结果
- 返回完整的变更列表和错误信息

### 11.3 状态对齐流程

```text
┌─────────────────┐
│ 解析 Manifest    │
│ (JSON + Schema)  │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 遍历 packages   │  ← 检测每个包的当前安装状态
│ 计算差异        │  ← installed/absent 与实际的差异
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 遍历 services   │  ← 检测每个服务的运行状态和自启状态
│ 计算差异        │  ← running/stopped/enabled 与实际的差异
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 遍历 configs    │  ← 读取当前配置，计算需要修改的键值
│ 计算差异        │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 遍历 users      │  ← 检测用户存在状态
│ 计算差异        │  ← present/absent 与实际的差异
└────────┬────────┘
         │
         ▼
┌─────────────────────────────────┐
│ dry-run: 输出差异列表            │
│ apply:  依次执行 install/remove/ │
│         start/stop/enable/       │
│         disable/config 操作      │
└─────────────────────────────────┘
```

### 11.4 API 接口

| 接口 | 方法 | 权限 | 说明 |
|------|------|------|------|
| `manifest_validate` | POST | `ssh.manage` | 校验 Manifest Schema |
| `manifest_dry_run` | POST | `ssh.manage` | 预演（只查看差异） |
| `manifest_apply` | POST | `ssh.manage` | 应用（真正执行） |

## 12. 异步任务队列

### 12.1 为什么需要异步

Host-Agent 使用单线程阻塞 TCP 服务器（`stream_socket_server` + `while` 循环）：

```php
while ($conn = @stream_socket_accept($server, -1)) {
    $response = host_agent_handle_request($request, $token, $root, $mode);
    fwrite($conn, $response);
    fclose($conn);  // ← 一次只处理一个连接
}
```

**耗时操作**（如 `apt-get install` 5-60s、`dnf upgrade` 30-300s）会阻塞整个 host-agent，导致：
- 后续请求排队等待
- 主应用端 curl 超时（默认 20s）
- 用户体验差

### 12.2 异步架构

```text
┌──────────────┐     POST /task/submit      ┌─────────────────────┐
│  主应用前端   │ ─────────────────────────→ │   host-agent        │
│              │  返回 {task_id, status}    │   (HTTP Server)     │
│              │                           │                     │
│              │  GET /task/status?id=xxx   │   ┌──────────────┐  │
│              │ ←──────────────────────── │   │  TaskQueue   │  │
│              │  返回 {status, result}    │   │  (内存数组)   │  │
└──────────────┘                           │   └──────────────┘  │
                                           │         │          │
                                           │    pcntl_fork      │
                                           │         ↓          │
                                           │   ┌──────────────┐  │
                                           │   │  Worker      │  │
                                           │   │  (子进程)     │  │
                                           │   └──────────────┘  │
                                           └─────────────────────┘
```

### 12.3 任务生命周期

```text
提交任务 → pending → running → completed/failed/cancelled
              ↑         ↑            ↑
           父进程记录  fork子进程    子进程写状态文件
           内存队列    执行任务      父进程waitpid回收
```

**核心函数：**

```php
host_agent_task_submit(string $action, array $payload): array
host_agent_task_status(string $taskId): array
host_agent_task_cancel(string $taskId): array
host_agent_task_list(): array
```

### 12.4 前端轮询机制

每个页面（packages.php / configs.php / manifests.php）都包含统一的 `pollTask()` 函数：

```javascript
async function pollTask(taskId, onProgress) {
    while (true) {
        await new Promise(resolve => setTimeout(resolve, 1500));
        const status = await apiPost('task_status', { task_id: taskId });
        if (onProgress) onProgress(status);
        if (status.status !== 'pending' && status.status !== 'running') {
            return status.result || status;
        }
    }
}
```

### 12.5 pcntl_fork 降级策略

当 PHP 未启用 `pcntl` 扩展时（常见于容器环境），自动降级为同步执行：

1. 仍然生成唯一的 `task_id`
2. 同步执行操作并将结果写入状态文件 `/tmp/host-agent-task-{taskId}.json`
3. 返回 `sync: true` 标记
4. 前端调用 `task_status` 时可直接从文件读取结果

### 12.6 API 接口

| 接口 | 方法 | 权限 | 说明 |
|------|------|------|------|
| `task_submit` | POST | `ssh.manage` | 提交异步任务 |
| `task_status` | GET/POST | `ssh.view`/`ssh.manage` | 查询任务状态 |
| `task_cancel` | POST | `ssh.manage` | 取消运行中任务 |

### 12.7 哪些操作已异步化

## 13. host 模式 vs simulate 模式完整对比

| 维度 | host 模式 | simulate 模式 |
|------|----------|--------------|
| **容器权限** | `Privileged: true` + `PidMode: host` | 普通容器 |
| **命令执行方式** | `nsenter -t 1 -m -u -i -n -p sh -lc $cmd` | `sh -lc $cmd`（容器内直接执行） |
| **根目录** | `/hostfs`（宿主机根目录挂载） | `/var/www/nav/data/host-agent-sim-root` |
| **包安装** | 真实安装到宿主机 | 在容器内执行，不改变宿主机 |
| **包卸载** | 真实从宿主机卸载 | 在容器内执行模拟命令 |
| **配置修改** | 修改宿主机真实配置文件 | 修改 sim-root 下的模拟文件 |
| **服务启停** | 真实 `systemctl start/stop/restart` | 模拟状态返回（不操作真实服务） |
| **进程查看** | 查看宿主机真实进程 | 查看容器内进程 |
| **用户管理** | `useradd`/`usermod` 操作宿主机 | 写入 sim-root 模拟状态文件 |
| **网络分析** | 查看宿主机网络状态 | 查看容器内网络状态 |
| **适用场景** | 生产环境真实运维 | 开发/测试环境安全演示 |
| **部署变量** | `HOST_AGENT_INSTALL_MODE=host`（默认） | `HOST_AGENT_INSTALL_MODE=simulate` |
| **Docker Compose** | 需挂载 `/var/run/docker.sock` | 需挂载 `/var/run/docker.sock` |
| **安全风险** | 高（可直接修改宿主机） | 低（隔离在容器内） |

### 13.1 模式切换方式

通过主应用容器的环境变量控制：

```yaml
# local/docker-compose.dev.yml
services:
  nav:
    environment:
      HOST_AGENT_INSTALL_MODE: "simulate"  # 或 "host"
```

安装时由 `host_agent_install_mode()` 读取该变量并传给 host-agent 容器：

```php
function host_agent_install_mode(): string {
    $mode = strtolower(trim((string)getenv('HOST_AGENT_INSTALL_MODE')));
    return in_array($mode, ['host', 'simulate'], true) ? $mode : 'host';
}
```

### 13.2 关键代码中的模式分支

```php
// cli/host_agent.php
function host_agent_host_shell(string $script, ?string $stdin = null): array {
    $mode = (string)(getenv('HOST_AGENT_MODE') ?: 'host');
    if ($mode === 'simulate') {
        return host_agent_proc_run(['sh', '-lc', $script], $stdin);
    }
    return host_agent_proc_run(
        ['/usr/bin/nsenter', '-t', '1', '-m', '-u', '-i', '-n', '-p', 'sh', '-lc', $script],
        $stdin
    );
}
```
