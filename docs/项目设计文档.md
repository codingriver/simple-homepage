## 一、项目概述

Simple Homepage 是一个面向个人、家庭网络、NAS、软路由、小型 VPS 的私有导航首页。采用 **PHP 8.2 + Nginx + JSON 文件存储** 架构，无需 MySQL/Redis，单容器即可运行。

### 1.1 技术栈

| 层级 | 技术 |
|------|------|
| 运行环境 | Docker (php:8.2-fpm-bookworm) |
| Web 服务器 | Nginx |
| 后端语言 | PHP 8.2 |
| 数据存储 | JSON 文件 (data/*.json) |
| 前端 | 原生 HTML/CSS/JS + 少量内联 PHP 渲染 |
| 自动化测试 | Playwright (TypeScript) + PHPUnit |
| 宿主桥接 | host-agent (PHP CLI HTTP Server) |

### 1.2 核心设计原则

1. **无数据库依赖**：所有配置、用户、站点数据均持久化为 JSON 文件。
2. **单容器可运行**：通过 Docker Compose 一键拉起。
3. **host-agent 解耦**：所有宿主机敏感操作（SSH、文件系统、进程、服务、Docker）均通过 host-agent 中转，不在 Web 进程内直接执行系统命令。
4. **文件型配置即代码**：Nginx 反代配置由后台根据站点数据自动生成。
5. **审计统一**：所有写操作（host-agent、SSH、文件、Docker）均进入审计日志。

## 五、共享核心模块（shared/）

该目录文件被 `public/`、`admin/`、`cli/`、`subsite-middleware/` 共同引用，是项目的"底层操作系统"。

### 5.1 auth.php（认证核心库）

**文件规模**：~1300 行  
**核心职责**：Token 生成验证、Cookie 管理、用户验证、IP 锁定、登录日志、首次安装检测、配置读取。

#### 关键函数分组

| 分组 | 代表函数 | 说明 |
|------|----------|------|
| 密钥管理 | `auth_secret_key()`, `auth_rotate_secret_key()` | 64B hex 随机密钥，持久化到 `data/auth_secret.key` |
| 配置读取 | `auth_get_config()`, `auth_default_config()` | 从 `config.json` 读取，缺失字段自动补默认值 |
| 安装检测 | `auth_needs_setup()`, `auth_apply_initial_install()` | `.installed` + `users.json` 双重判断 |
| Token | `auth_generate_token()`, `auth_verify_token()` | JWT-like HMAC-SHA256，含 jti 防重放 |
| 用户管理 | `auth_load_users()`, `auth_save_user()`, `auth_check_password()` | JSON 文件存储，bcrypt 哈希（cost=10） |
| 权限系统 | `auth_user_permissions()`, `auth_user_has_permission()` | 角色权限 + 自定义权限混合模型 |
| 开发模式 | `auth_dev_mode_enabled()`, `auth_dev_qa_user_record()` | 注入内存虚拟用户 `qatest`，不落盘 |
| IP 锁定 | `ip_locks_load()`, `ip_locks_save()` | 自动清理过期记录 |
| 客户端 IP | `get_client_ip()` | 支持 X-Real-IP、X-Forwarded-For 链、REMOTE_ADDR |
| Cookie 策略 | `auth_cookie_domain_for_request()`, `auth_cookie_secure_for_request()` | IP 访问自动降级，HTTPS 自适应 |
| 重定向安全 | `auth_sanitize_redirect()` | 防开放重定向，仅允许本站群地址 |

### 5.2 notify_runtime.php（统一通知运行时）

**核心职责**：通知渠道配置持久化、事件定义、Webhook 构造与发送、冷却控制。

**支持渠道**：Telegram、飞书、钉钉、企业微信、自定义 Webhook。

**关键函数**：
- `notify_load_data()` / `notify_save_data()` —— `data/notifications.json`
- `notify_channel_upsert()` / `notify_channel_delete()` / `notify_channel_toggle()`
- `notify_build_request()` —— 按渠道类型构造请求体
- `notify_send_channel()` —— 单渠道发送并记录运行时状态
- `notify_event()` —— 广播事件到所有匹配渠道
- `notify_test_channel()` —— 测试发送

### 5.3 request_timing.php（请求耗时日志）

**核心职责**：在每个 PHP 请求收到时和响应结束时写入耗时日志。

**日志路径**：`data/logs/request_timing.log`

**控制方式**：
- `NAV_REQUEST_TIMING=0`：关闭
- `NAV_REQUEST_TIMING_CLI=1`：CLI 模式也开启

**实现机制**：文件加载时自动写 `recv` 行，`register_shutdown_function` 写 `done` 行。

## 七、子站中间件（subsite-middleware/）

### 7.1 auth_check.php

**定位**：供其他 PHP 子站在入口文件顶部 `require_once` 使用的统一鉴权中间件。

**工作流程**：

```text
1. 定位并加载 shared/auth.php
2. auth_get_current_user() 获取当前用户
3. 未登录 → 302 跳转到导航站登录页（携带 redirect=当前URI）
4. 已登录 → 将 $user 存入 $GLOBALS['nav_user'] 供子站读取
```

> 注意：`_nav_token` 的 Cookie 写入和 URL 清洗逻辑存在已知的代码结构问题（位于未登录分支的 exit 之后，常规执行流中不可达）。这是当前代码中的已知缺陷。

## 九、host-agent 架构

### 9.1 为什么需要 host-agent

Web 进程运行在容器内，若需操作宿主机文件系统、SSH、进程、服务、Docker，不能直接在 PHP-FPM 中执行特权命令。host-agent 作为独立容器/进程，通过 HTTP API 提供统一抽象。

### 9.2 调用链

```text
admin/xxx.php
  → admin/xxx_api.php
    → admin/shared/host_agent_lib.php
      → host-agent HTTP API (cli/host_agent.php)
        → 实际系统操作（host 模式）或模拟文件操作（simulate 模式）
```

### 9.3 运行模式

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| `host` | host-agent 以特权容器运行，挂载 `/hostfs`，可操作真实宿主机 | 生产环境 |
| `simulate` | 操作落在 `data/host-agent-sim-root/` 下，不碰真实宿主机 | 开发/测试 |

### 9.4 host-agent API 分类

- **SSH 管理**：`/ssh/status`, `/ssh/config`, `/ssh/action`, `/ssh/config/validate`, `/ssh/enable`, `/ssh/install`
- **远程主机**：`/remote/test`, `/remote/exec`
- **文件系统**：`/fs/list`, `/fs/read`, `/fs/write`, `/fs/delete`, `/fs/mkdir`, `/fs/rename`, `/fs/copy`, `/fs/move`, `/fs/search`, `/fs/stat`, `/fs/chmod`, `/fs/chown`, `/fs/chgrp`, `/fs/acl/apply`, `/fs/archive`, `/fs/extract`
- **终端**：`/terminal/open`, `/terminal/read`, `/terminal/write`, `/terminal/close`
- **系统概览**：`/system/overview`
- **进程/服务**：`/process/list`, `/process/kill`, `/service/list`, `/service/action`, `/service/logs`
- **网络**：`/network/overview`
- **Docker**：`/docker/summary`, `/docker/containers`, `/docker/container/action`, `/docker/images`, `/docker/volumes`, `/docker/networks`
- **用户/用户组**：`/user/list`, `/user/save`, `/user/delete`, `/user/password`, `/user/lock`, `/group/list`, `/group/save`, `/group/delete`
- **共享服务**：`/share/sftp/*`, `/share/smb/*`, `/share/ftp/*`, `/share/nfs/*`, `/share/afp/*`, `/share/async/*`

## 十、安全设计

### 10.1 认证安全

- Token：HMAC-SHA256 签名，含 `jti` 随机唯一 ID，防重放。
- Cookie：HttpOnly + SameSite=Lax；Secure 根据请求协议和配置自适应。
- 密码：bcrypt 哈希，cost=10。
- IP 锁定：连续失败 5 次锁定 15 分钟。
- 会话撤销：支持通过 `data/sessions.json` 按 jti 撤销特定会话。

### 10.2 访问控制

- `data/` 目录下 `.json` 文件禁止 Web 直接访问（由 Nginx 配置控制）。
- `setup.php` 安装完成后返回 404。
- 所有写操作 POST 表单必须经过 CSRF Token 校验。
- CLI 脚本 Web 访问返回 403。

### 10.3 输入安全

- 代理目标 `proxy_target` 正则白名单限制，仅允许 `http(s)://host:port` 格式，防止 Nginx 配置注入。
- Favicon 抓取禁止内网 IP、`.local` 等私有后缀。
- 背景图文件名经过 `basename + preg_replace` 清洗，防路径遍历。
- 备份文件名正则校验 `^backup_[\d_a-z]+\.json$`。
- 文件系统操作统一走 host-agent，路径经过归一化和白名单校验。

### 10.4 子站 SSO

- 配置 `cookie_domain=.yourdomain.com` 后，导航站与子站共享 Cookie，实现单点登录。
- 子站通过 `_nav_token` URL 参数首次传递 Token，中间件自动写入 Cookie 并清洗 URL（当前存在实现缺陷，见第七章）。

## 十二、关键数据流

### 12.1 登录流程

```text
POST login.php
  → csrf_check()
  → ip_locks_load() 检查锁定
  → auth_check_password() 验证密码
  → auth_generate_token() 生成 Token
  → auth_set_cookie() 写入 Cookie
  → auth_write_log('SUCCESS') + webhook_send()
  → 跳转首页或 redirect
```

### 12.2 保存代理站点 → Nginx 配置生效流程

```text
admin/sites.php POST save
  → save_sites() 写入 sites.json
  → 页面提示保存成功
  → 首页检测 homepage_pending_proxy_sites() 显示未生效横幅
  → admin/settings.php POST nginx_apply_and_reload
    → nginx_generate_proxy_conf() 生成 nav-proxy.conf / nav-proxy-domains.conf
    → admin_run_command('nginx -t && nginx -s reload')
    → 成功则更新 config.json nginx_last_applied
```

### 12.3 DDNS 执行流程

```text
cron 触发 cli/ddns_sync.php
  → ddns_due_tasks() 读取到期任务
  → ddns_run_task($task)
    → 获取当前公网 IP（多来源 + 线路优选 + fallback）
    → dns_api_upsert() 调用 DNS 提供商 API
    → 更新 ddns_tasks.json runtime 状态
    → 写入 data/logs/ddns_{id}.log
  → 汇总退出码（0=全部成功，1=任一失败）
```

### 12.4 通知发送流程

```text
业务代码调用 notify_event('task_failed', [...])
  → notify_load_data() 读取 channels
  → 遍历 enabled 且订阅了该事件的渠道
  → notify_channel_is_due() 冷却检查
  → notify_send_channel()
    → notify_build_request() 按渠道类型构造 payload
    → notify_http_post_json() 发送
    → notify_channel_mark_sent() 更新运行时状态
    → notify_log_write() 写日志
```
