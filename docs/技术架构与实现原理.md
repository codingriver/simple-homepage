# RiverOps 技术架构与实现原理

本文档描述当前 RiverOps 运维控制台的实际架构与系统边界。

## 1. 系统边界

RiverOps 面向个人、家庭网络、NAS、软路由和小型 VPS，集中提供：

- 用户、权限、会话与 API Token 管理
- DNS、DDNS 与域名有效期监控
- Shell、PHP、Python、Node.js 和自定义计划任务
- Node.js/npm 运行环境管理
- Nginx、PHP-FPM 与 PHP 配置只读查看
- 配置备份、恢复、Webhook、日志与调试工具

根路径 `/` 只负责跳转至 `/admin/index.php`，公共页面仅包含安装、登录、退出和 Nginx `auth_request` 鉴权入口。

## 2. 容器架构

```text
Browser
   │
   ▼
Nginx :${RIVEROPS_PORT:-58080}
   ├── static files
   ├── /auth/verify.php ── auth_request
   └── PHP requests ────── Unix socket /run/nginx/php-fpm.sock
                              │
                              ▼
                         PHP-FPM 8.2
                              │
                              ▼
                  /var/www/riverops/data
```

容器由 runit 同时管理 Nginx、PHP-FPM 和 Cron。Web 进程统一使用 Linux 用户和组 `riverops`，默认 UID/GID 为 1000，启动时可通过 `PUID`、`PGID` 或数据目录 owner 对齐。

项目代码位于 `/var/www/riverops`，持久化数据挂载到 `/var/www/riverops/data`。

## 3. 目录职责

```text
public/       安装、登录、退出、根路径跳转和 auth_request 入口
admin/        后台页面、AJAX API 与业务领域共享库
shared/       认证、HTTP 客户端、请求耗时等核心共享能力
cli/          计划任务、DDNS、域名有效期、运行环境任务和用户管理脚本
python/       DNS 辅助逻辑
docker/       Entrypoint、Nginx、PHP-FPM、PHP 和 runit 配置
nginx-conf/   容器站点配置
data/         JSON 数据、日志、任务、备份和持久化运行配置
tests/        PHPUnit、Playwright 与测试辅助工具
local/        本地 Compose、开发覆盖和构建脚本
```

## 4. 请求生命周期

### 4.1 公共请求

1. `/setup.php` 仅在尚未安装时开放。
2. `/login.php` 负责 CSRF、密码校验、IP 锁定、设备会话上限和 Cookie 写入。
3. 登录成功后先进入同源完成页，确保 `riverops_session` 稳定写入，再跳转后台。
4. `/auth/verify.php` 供 Nginx `auth_request` 调用，成功时返回用户和角色响应头。

### 4.2 后台页面

1. 页面加载 `admin/shared/functions.php`。
2. 所有 POST 处理必须在输出模板之前完成。
3. `admin/shared/header.php` 校验权限、生成 CSRF Token 并释放 PHP Session 锁。
4. 页面输出 HTML，最后加载 `admin/shared/footer.php`。
5. POST 成功后使用 PRG 跳转，避免重复提交。

### 4.3 AJAX API

- 以 `declare(strict_types=1);` 开头。
- 要求 `X-Requested-With: XMLHttpRequest`。
- 写操作必须验证 CSRF。
- 返回 `ok`、`msg`、`data` 组成的 JSON。
- 未知 action 返回 404 JSON，鉴权失败返回 401 JSON。

## 5. 认证和安全

- 登录 Cookie：`riverops_session`
- PHP Session Cookie：`riverops_php_session`
- 会话状态：`data/sessions.json`
- 用户数据：`data/users.json`
- Token 密钥：`data/auth_secret.key`
- IP 锁定：`data/ip_locks.json`

Cookie 支持 Domain Cookie 与 host-only Cookie 同名共存，服务端解析原始 Cookie 头并逐个验证候选 Token。会话文件读操作使用共享锁，注册、撤销、清理和 touch 使用独占锁。

用户配置域名字段为 `panel_domain`。旧字段、旧 Cookie、旧环境变量和旧容器路径不再读取。

HTTP 客户端统一经过 SSRF 防护；文件操作必须使用白名单、`basename()` 或 `realpath()` 边界校验；HTML 输出必须转义。

## 6. 数据存储

RiverOps 不依赖数据库，业务状态存储为 JSON 文件。主要文件包括：

```text
config.json
users.json
sessions.json
api_tokens.json
scheduled_tasks.json
dns_config.json
ddns_tasks.json
domain_expiry.json
notifications.json
```

写入使用 `LOCK_EX`，高频共享文件使用显式 `flock()` 完成读改写事务。运行日志、任务目录、备份和持久化服务配置均位于 `data/`。

## 7. 计划任务与运行环境

数字任务使用 `data/tasks/task_{id}/`：

| 类型 | 入口 |
|---|---|
| Shell | `run.sh` |
| PHP | `main.php` |
| Python | `main.py` |
| Node.js | `main.mjs` |

执行器具有硬超时、PID 锁、僵尸锁清理和独立日志。Node.js 与 Python 依赖只能安装在任务目录内。

Node.js 多版本位于 `data/runtime/node/versions/`，当前版本由 `data/runtime/node/current` 指向。安装任务由 CLI 后台进程执行，状态与日志写入 `data/runtime/jobs/`。

## 8. 持久化运行配置

Entrypoint 首次启动时生成：

```text
data/nginx/nginx.conf
data/nginx/http.d/riverops.conf
data/php-fpm/riverops.conf
data/php/custom.ini
```

容器内对应文件为：

```text
/etc/nginx/nginx.conf
/etc/nginx/http.d/riverops.conf
/usr/local/etc/php-fpm.d/riverops.conf
/usr/local/etc/php/conf.d/99-riverops-custom.ini
```

后台只读查看这些配置，不执行保存或 Reload。修改持久化文件后必须重启容器。

## 9. 环境变量

| 变量 | 说明 |
|---|---|
| `RIVEROPS_PORT` | 容器监听端口，默认 58080 |
| `RIVEROPS_DEV_MODE` | 开发模式开关 |
| `RIVEROPS_REQUEST_TIMING` | 请求耗时日志开关 |
| `RIVEROPS_REQUEST_TIMING_CLI` | CLI 请求耗时日志开关 |
| `PUID` / `PGID` | 容器运行用户 UID/GID |
| `ADMIN` / `PASSWORD` / `NAME` / `DOMAIN` | 无人值守首次安装 |
| `HTTP_PROXY` / `HTTPS_PROXY` / `NO_PROXY` | 出站代理 |

旧品牌环境变量没有别名或回退逻辑。

## 10. 测试分层

- PHPUnit：核心认证、后台共享库、CLI、公共入口和 Docker 配置。
- Playwright：桌面 Chromium 与 Pixel 7 移动端后台流程。
- Lighthouse CI：登录和后台入口性能、可访问性与最佳实践。
- Docker 验证：Compose 解析、镜像构建、Nginx/PHP-FPM 配置和容器冒烟。

涉及认证、Cookie、容器路径、计划任务或数据锁的修改，必须同时补充对应单元测试与 E2E 回归。
