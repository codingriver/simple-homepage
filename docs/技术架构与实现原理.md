## 整体架构

```text
┌─────────────────────────────────────────────────────────────┐
│                         用户浏览器                           │
└─────────────────────────┬───────────────────────────────────┘
                          │
┌─────────────────────────▼───────────────────────────────────┐
│  前置 Nginx / Caddy / Traefik（可选，域名+HTTPS场景）        │
└─────────────────────────┬───────────────────────────────────┘
                          │

│  │  PHP-FPM（navwww 用户运行）                            │    │
│  │  ├── public/   前台入口                                │    │

│  │  └── subsite-middleware/ 子站鉴权中间件                │    │

│  ┌─────────────────────────────────────────────────────┐    │
│  │  Supervisor（PID 1）                                   │    │
│  │  └── 同时管理 Nginx + PHP-FPM + Cron                  │    │
```

### 架构选型说明

项目采用**单容器 Debian 方案**（`php:8.2-fpm-bookworm` + Nginx + Supervisor），核心原因：

- 导航站属于轻量级应用，无需微服务分离
- Debian 系对常见第三方 Linux 二进制兼容性更好
- Supervisor 统一管理进程，崩溃自动重启
- 一个 `docker compose up` 即可拉起，运维简单

## 请求生命周期

### 前台请求

```text
浏览器请求 /index.php
    ↓
Nginx（监听 NAV_PORT，默认 58080）
    ↓
fastcgi_pass unix:/run/php-fpm.sock
    ↓
PHP-FPM worker 加载 public/index.php
    ├── shared/auth.php → auth_check_setup() / auth_get_current_user()
    ├── 读取 sites.json → 过滤可见分组
    ├── 读取 config.json → 获取站点名称、背景、卡片样式
    ├── 可选：读取 health_cache.json → 站点健康状态
    └── 渲染 HTML + 内联 CSS/JS
```

### 后台请求

```text
浏览器请求 /admin/settings.php
    ↓
Nginx location ^~ /admin/ { auth_request /auth/verify.php; }
    ↓
auth_request 子请求 → public/auth/verify.php
    ├── 验证 Cookie 中的 Token
    └── 返回 200 或 401
    ↓
主请求继续 → admin/settings.php
    ├── auth_request off（PHP location 内关闭，避免死锁）
    ├── shared/auth.php → 二次验证 + 获取当前用户
    ├── admin/shared/functions.php → 加载配置/站点/备份逻辑
    └── 渲染后台页面
```

### 关键安全点

- `auth_request` 子请求与主请求可能争抢 PHP-FPM worker 导致 504 死锁
- **修复**：PHP location 内加 `auth_request off;`，由 PHP 自行鉴权

## 认证与安全模型

### Token 机制

项目采用 **JWT-like Token**（HMAC-SHA256 签名），不是标准 JWT，但设计理念类似：

```php
$payload = [
    'username' => $username,
    'role'     => $role,
    'iat'      => time(),
    'exp'      => time() + $ttl,
    'jti'      => bin2hex(random_bytes(16)),  // 防重放
];
$token = base64_encode(json_encode($payload)) . '.' . hash_hmac('sha256', ..., AUTH_SECRET_KEY);
```

**特点：**

- `jti`（JWT ID）每次登录唯一，支持会话级撤销
- 撤销列表保存在 `data/sessions.json`
- 有效期默认 8 小时，"记住我"延长至 60 天

### Cookie 策略

```php
setcookie(SESSION_COOKIE_NAME, $token, [
    'expires'  => $expire,
    'path'     => '/',
    'domain'   => $cookie_domain,
    'secure'   => $secure,      // off / auto / on
    'httponly' => true,
    'samesite' => 'Lax',
]);
```

**Secure 三档模式：**

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| `off` | HTTP/HTTPS 均可发送 | 内网调试、IP 访问 |
| `auto` | 自动检测协议（含 X-Forwarded-Proto）| VPS 反代、混合环境 |
| `on` | 仅 HTTPS 发送 | 生产环境全程 HTTPS |

**IP 访问自动降级：** 当 `HTTP_HOST` 为 IP 格式时，代码自动设置 `secure=false, domain=空`，确保内网应急访问始终可用。

### CSRF 防护

```php
function csrf_token(): string {
    if (session_status() === PHP_SESSION_NONE) session_start();
    if (empty($_SESSION['csrf_token'])) {
        $_SESSION['csrf_token'] = bin2hex(random_bytes(32));
    }
    return $_SESSION['csrf_token'];
}

function csrf_check(): void {
    $token = $_POST['_csrf'] ?? '';
    $expected = $_SESSION['csrf_token'] ?? '';
    if (!$token || !$expected || !hash_equals($expected, $token)) {
        http_response_code(403);
        exit('CSRF验证失败');
    }
}
```

- 所有修改数据的操作（包括登录、退出、安装向导）都必须验证 CSRF Token
- AJAX 鉴权失败返回 JSON（401），不能 302 重定向

### IP 锁定

连续登录失败 5 次后，自动锁定该 IP 15 分钟。记录保存在 `data/ip_locks.json`，每次登录时自动清理过期记录。

### 密码安全

```php
$hash = password_hash($password, PASSWORD_BCRYPT, ['cost' => 10]);
password_verify($input, $hash);
```

### 密钥管理

`AUTH_SECRET_KEY` 优先级：

1. 环境变量 `AUTH_SECRET_KEY`
2. `data/auth_secret.key`（64B hex 随机字符串，权限 600）
3. 安装向导自动生成

## Host-Agent 架构

### 为什么需要 Host-Agent

Web 进程运行在容器内，若需操作宿主机文件系统、SSH、进程、服务，不能直接在 PHP-FPM 中执行特权命令。Host-Agent 作为独立容器/进程，通过 HTTP API 提供统一抽象。

### 调用链

```text
admin/xxx.php
  → admin/xxx_api.php
    → admin/shared/host_agent_lib.php
      → host-agent HTTP API (cli/host_agent.php)
        → 实际系统操作（host 模式）或模拟文件操作（simulate 模式）
```

### 运行模式

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| `host` | 特权容器运行，挂载 `/hostfs`，可操作真实宿主机 | 生产环境 |
| `simulate` | 操作落在 `data/host-agent-sim-root/` 下 | 开发/测试（默认） |

### Host-Agent API 分类

- **SSH 管理**：`/ssh/status`, `/ssh/config`, `/ssh/action`, `/ssh/config/validate`, `/ssh/enable`, `/ssh/install`
- **文件系统**：`/fs/list`, `/fs/read`, `/fs/write`, `/fs/delete`, `/fs/mkdir`, `/fs/rename`, `/fs/copy`, `/fs/move`, `/fs/search`, `/fs/stat`, `/fs/chmod`, `/fs/chown`, `/fs/chgrp`, `/fs/acl/apply`, `/fs/archive`, `/fs/extract`
- **终端**：`/terminal/open`, `/terminal/read`, `/terminal/write`, `/terminal/close`
- **系统概览**：`/system/overview`
- **进程/服务**：`/process/list`, `/process/kill`, `/service/list`, `/service/action`, `/service/logs`
- **网络**：`/network/overview`
- **Docker**：`/docker/summary`, `/docker/containers`, `/docker/container/action`, `/docker/images`, `/docker/volumes`, `/docker/networks`
- **用户/用户组**：`/user/list`, `/user/save`, `/user/delete`, `/user/password`, `/user/lock`, `/group/list`, `/group/save`, `/group/delete`
- **共享服务**：`/share/sftp/*`, `/share/smb/*`, `/share/ftp/*`, `/share/nfs/*`, `/share/afp/*`, `/share/async/*`

### 安全设计

- 页面层不直接访问 `docker.sock`
- Docker API 或 `docker` 命令统一由 host-agent 执行
- 所有写操作都进入审计
- 开发/测试默认使用 `simulate` 模式，避免误改宿主机

## Nginx 反向代理自动生成

### 设计目标

后台配置的 Proxy 类型站点，需要自动生成 Nginx 反代配置，无需用户手动编辑 Nginx 配置文件。

### 生效流程

```text
admin/sites.php POST save
  → save_sites() 写入 sites.json
  → 页面提示保存成功
  → 首页检测 homepage_pending_proxy_sites() 显示"未生效"横幅
  → admin/settings.php POST nginx_apply_and_reload
    → nginx_generate_proxy_conf() 生成 nav-proxy.conf / nav-proxy-domains.conf
    → admin_run_command('nginx -t && nginx -s reload')
    → 成功则更新 config.json 中的 nginx_last_applied
```

### 两种代理模式

**路径模式**：`https://nav.example.com/p/{slug}/`

```nginx
location ^~ /p/{slug}/ {
    proxy_pass http://{proxy_target}/;
    include /etc/nginx/proxy_params_full.conf;
}
```

**子域模式**：`https://{proxy_domain}/`

```nginx
server {
    listen 58080;
    server_name {proxy_domain};
    location / {
        proxy_pass http://{proxy_target}/;
        include /etc/nginx/proxy_params_full.conf;
    }
}
```

### 安全校验

`proxy_target` 正则白名单限制，仅允许 `http(s)://host:port` 格式，防止 Nginx 配置注入。同时禁止 SSRF：目标地址不能是内网回环地址或私有后缀（如 `.local`）。

## 子站 SSO 机制

### 目标

让导航站登录后，子站（同域名下的其他应用）无需再次登录。

### 实现方式

1. **Cookie 共享**：配置 `cookie_domain=.yourdomain.com` 后，导航站与子站共享 Cookie
2. **Token 传递**：子站首次通过 `_nav_token` URL 参数接收 Token
3. **中间件写入**：`subsite-middleware/auth_check.php` 自动将 Token 写入 Cookie 并清洗 URL

### 子站接入方式

**PHP 子站**：

```php
<?php
require_once '/var/www/nav/subsite-middleware/auth_check.php';
echo '当前用户：' . $nav_user['username'];
```

**Nginx auth_request**：

```nginx
location /protected/ {
    auth_request /auth/verify.php;
    proxy_pass http://backend/;
}
```

## 通知与 Webhook 架构

### 通知运行时

核心文件：`shared/notify_runtime.php`

**支持渠道**：Telegram、飞书、钉钉、企业微信、自定义 Webhook。

### 事件订阅

在 `admin/notifications.php` 中配置渠道时，可订阅以下事件：

- `SUCCESS` — 登录成功
- `FAIL` — 登录失败
- `IP_LOCKED` — IP 被锁定
- `LOGOUT` — 用户登出
- `SETUP` — 完成安装向导
- `task_failed` — 计划任务失败

### 发送流程

```text
业务代码调用 notify_event('task_failed', [...])
  → notify_load_data() 读取 channels
  → 遍历 enabled 且订阅了该事件的渠道
  → notify_channel_is_due() 冷却检查（防止轰炸）
  → notify_send_channel()
    → notify_build_request() 按渠道类型构造 payload
    → notify_http_post_json() 发送
    → notify_channel_mark_sent() 更新运行时状态
    → notify_log_write() 写日志
```

### 日志位置

`data/logs/notify_runtime.log`

## DDNS 执行流程

### 触发方式

1. **手动执行**：后台 DDNS 页面点击"立即执行"
2. **定时执行**：Cron 定时调用 `cli/ddns_sync.php`

### 执行流程

```text
cron 触发 cli/ddns_sync.php
  → ddns_due_tasks() 读取到期任务
  → ddns_run_task($task)
    → 获取当前公网 IP（多来源 + 线路优选 + fallback）
    → dns_api_upsert() 调用 DNS 提供商 API
    → 更新 ddns_tasks.json runtime 状态
    → 写入 data/logs/ddns_{id}.log
  → 汇总退出码（0=全部成功，1=任一失败）
```

### IP 来源策略

支持多个公网 IP 检测源，按优先级和响应速度自动选择。如果主源失败，自动 fallback 到备用源。

## 计划任务调度原理

### 数据模型

每个任务包含：

```json
{
  "id": "task_id",
  "name": "任务名称",
  "script_filename": "task_id.sh",
  "schedule": "*/5 * * * *",
  "command": "echo hello",
  "enabled": true,
  "last_run": "2026-04-15 10:00:00",
  "last_code": 0,
  "last_output": "hello"
}
```

### 脚本落地

保存任务时，命令文本会被写入文件：

```text
data/tasks/task_id.sh
```

同时 Cron 调度表会同步更新（通过 `cron_regenerate()`）。

### 执行方式

1. **Cron 自动执行**：容器内 crond 按 schedule 触发
2. **后台立即执行**：点击"立即执行"，由 `cli/run_scheduled_task.php` 处理

### 日志文件

每个任务的日志保存在同名文件：

```text
data/tasks/task_id.log
```

### 调度器保护

防止多个执行实例同时运行同一个任务。通过锁文件机制实现。

## 安全注意事项

1. **认证机制**：JWT-like Token + Cookie HttpOnly + SameSite=Lax + 会话级撤销
2. **IP 锁定**：登录失败超过限制后自动锁定 IP
3. **Host-Agent 模式**：开发/测试环境强烈建议使用 `simulate` 模式
4. **docker.sock 挂载**：仅在一键安装/升级 host-agent 时临时挂载，完成后建议移除
5. **SSRF 防护**：所有根据用户输入发起外部 HTTP 请求的代码必须经过目标地址安全校验
6. **文件路径安全**：禁止直接使用用户输入拼接路径，优先使用 `basename()` 或正则白名单
