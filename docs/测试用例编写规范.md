# 测试用例编写规范

> 面向 Agent / AI 工具的界面测试用例编写规范。  
> 目标：强制测试编写过程标准化、结构化、可追溯，避免遗漏关键测试维度。  
> 适用范围：任意 Web / App / 管理后台 / 控制台类界面测试设计任务。  
> 与具体项目无关；本规范仅约束“如何完整编写界面测试用例”。

---

## 一、核心目标

Agent 在为任意界面编写测试用例时，**不得只围绕按钮点击或单一路径进行编写**，必须按照统一维度逐项检查并覆盖。

每一个界面都必须先回答以下问题：

1. 该界面的业务目标是什么。
2. 编写依据是什么，哪些信息来自需求/原型/代码/现有页面，哪些信息尚未确认。
3. 该界面有哪些可见元素与可操作元素。
4. 用户的主路径是什么。
5. 该界面有哪些异常、边界、权限、状态与响应式风险。
6. 哪些维度适用，哪些维度不适用，哪些维度因信息不足而暂无法下结论。
7. 对于不适用维度，必须明确写：`无需`，并说明原因；对于信息不足项，必须写：`待确认`，并说明缺失信息。

---

## 二、总原则

### 1. 禁止遗漏维度

Agent 编写界面测试用例时，必须逐项检查本规范定义的所有维度；**未检查某维度，不得默认视为“不存在”**。

### 2. 禁止凭经验跳项

即使某页面看起来简单，也必须先按完整维度过一遍，再决定：
- 需要测试
- 可合并测试
- 无需测试

### 3. 禁止只测主流程

主流程测试是必要项，但不是充分项。必须同时考虑：
- 异常路径
- 权限路径
- 状态变化
- 边界条件
- 响应式表现
- 数据一致性

### 4. 禁止把“未发现”写成“无需”

只有在明确确认该界面不存在某类能力或场景时，才允许写：`无需`。并且必须附带原因。

错误写法：
- 无需测试分页

正确写法：
- 无需测试分页；原因：该界面无列表、无分页控件，且数据不按页展示。

### 5. 禁止只验证静态显示

界面测试必须覆盖：
- 显示是否正确
- 交互是否可用
- 状态是否变化
- 数据是否真实落地或真实生效

### 6. 禁止只做前端表象验证

只验证 toast、文案、弹窗显示不够；还必须根据界面能力确认：
- 列表是否更新
- 详情是否更新
- 刷新后是否保持
- 后端结果是否一致

### 7. 禁止臆造页面、控件、权限或业务规则

Agent 只能根据以下信息来源编写测试：
- 用户明确提供的需求与说明
- 项目文档
- 可确认的页面实现
- 已存在的业务规则或测试代码

若没有证据，不得自行假设：
- 一定存在某个按钮
- 一定存在某个角色
- 一定存在某个接口或校验规则
- 一定存在某个跳转、弹窗、分页、导出、二次确认

### 8. 信息不足时必须写 `待确认`，不得偷换成 `无需`

`无需` 只用于“明确不存在”。  
若只是“目前没看到”或“文档未说明”，必须写：
- `待确认 <维度/能力>；原因：当前提供的资料中未说明，且无法从现有页面/代码明确确认`

错误写法：
- 无需测试权限；原因：文档里没写

正确写法：
- 待确认权限差异；原因：当前资料只说明管理员视角，未说明是否存在普通用户、只读角色或匿名访问差异。

### 9. 必须先拆页面，再拆页面内功能，再拆交互动作

Agent 不得直接按“整个系统”一把写测试点。  
必须先按以下层级拆分：
1. 页面/路由/主界面
2. 页面内功能区块
3. 区块内具体能力
4. 每个能力下的正常/异常/边界/权限/状态/响应式测试点

### 10. 跨页面链路必须拆解，不得混写成一团

如果一个业务流程跨多个页面，必须拆成：
- 起始页面
- 跳转动作
- 目标页面
- 返回链路
- 数据回写或状态同步点

不得把“登录页 -> 列表页 -> 详情页 -> 编辑弹窗 -> 返回首页”直接写成一个无边界的大而全测试块，而不说明每一步分别属于哪个界面。

---

## 三、标准编写流程

Agent 在开始编写任意界面测试用例前，必须按以下顺序进行分析：

1. **确认输入依据与任务范围**
2. **列出本次任务涉及的页面清单**
3. **标记哪些页面在范围内，哪些页面不在范围内**
4. **对每个页面拆分功能区块与交互能力**
5. **明确页面目标**
6. **列出界面元素**
7. **梳理主流程**
8. **梳理异常与边界**
9. **梳理状态变化**
10. **梳理权限与身份差异**
11. **梳理响应式与终端差异**
12. **梳理前后端一致性验证点**
13. **逐项标注“需要 / 无需 / 待确认”并写明原因**
14. **再输出测试用例清单**

未完成上述分析，不得直接开始写测试代码或测试用例。

### 页面拆分的最低要求

若任务涉及多个页面，必须先输出页面清单。页面清单至少应包含：
- 页面名称
- 页面路径或入口
- 页面类型
- 本次是否纳入测试范围
- 不纳入范围的原因

### 页面内功能拆分的最低要求

每个页面至少要继续拆到“功能区块/能力”层级，例如：
- 页面：用户管理
- 功能区块：用户列表
- 能力：查看列表、搜索用户、创建用户、编辑角色、删除用户

若页面存在全局区域，也必须单独识别是否属于页面测试范围，例如：
- 顶部导航
- 侧边栏
- 全局搜索
- 全局通知
- 通用弹窗
- 上传/下载组件

### 超出范围处理

若发现某能力明显属于其他页面或其他系统，不得强行纳入当前界面测试。必须写：
- `不在本次范围；原因：<说明其真实归属页面/模块/系统>`

### 信息不足处理

若任务资料不足以支持完整拆分，必须先输出缺失项，例如：
- 缺少页面原型
- 缺少角色说明
- 缺少接口或数据规则
- 缺少移动端是否支持的说明

在缺失信息未被补齐前，不得把不确定内容写成确定测试点。

---

## 四、界面测试维度清单（强制逐项检查）

以下每一个维度，Agent 都必须判断：
- 是否适用
- 如果适用，应测试什么
- 如果不适用，必须写 `无需` + 原因
- 如果当前无法确认，必须写 `待确认` + 原因

---

### A. 页面目标与定位

#### 必查内容
- 该界面的核心业务目标
- 用户进入该界面后最主要的任务
- 页面属于以下哪类：
  - 展示类
  - 表单类
  - 列表类
  - 详情类
  - 操作类
  - 配置类
  - 审批类
  - 日志/监控类
  - 混合型页面

#### 输出要求
必须写明：
- 页面用途
- 用户目标
- 核心成功标准

#### 如不适用
- 不允许写无需
- 原因：每个界面都必须有目标定义

---

### B. 基础可见元素

#### 必查内容
- 页面标题
- 副标题 / 说明文案
- 主要区域块
- 列表 / 表格 / 卡片 / 表单
- 按钮
- 输入框
- 下拉框
- 开关 / 单选 / 复选
- 弹窗 / 抽屉 / 浮层
- 提示文案 / 帮助文案
- 空状态提示
- 错误提示区域

#### 应测试内容
- 是否显示
- 文案是否正确
- 默认值是否正确
- 是否可操作
- 是否与页面当前状态一致

#### 可写无需的条件
若页面确实不存在某类元素，可写：
- 无需测试弹窗；原因：该页面无弹窗、抽屉或浮层交互。
- 无需测试输入框默认值；原因：该页面为纯展示页，无任何输入控件。

---

### C. 页面访问与入口

#### 必查内容
- 是否可正常进入页面
- 页面入口是否存在
- 页面跳转是否正确
- 刷新后是否可正常恢复
- 直接访问 URL 是否正常

#### 应测试内容
- 从正常入口进入
- 通过直接链接进入
- 页面初次加载是否成功
- 非法入口或缺失前置条件时是否处理正确

#### 可写无需的条件
- 无需测试页面入口按钮；原因：该页面无独立入口，仅作为嵌入式组件存在。

---

### D. 主流程

#### 必查内容
围绕用户最主要任务设计完整路径，例如：
- 查看
- 新建
- 编辑
- 删除
- 提交
- 搜索
- 过滤
- 执行
- 下载
- 上传
- 导入
- 导出

#### 应测试内容
至少覆盖：
1. 正常进入主流程
2. 完成关键输入或关键操作
3. 成功提交或成功执行
4. 界面反馈正确
5. 数据结果正确
6. 刷新后结果仍正确

#### 可写无需的条件
若页面无创建/修改/执行类能力，则可写：
- 无需测试新建流程；原因：该页面为只读展示页，无创建能力。
- 无需测试删除流程；原因：该页面不提供删除操作。

---

### E. 表单输入与校验

#### 必查内容
只要页面存在输入控件，就必须检查：
- 必填项
- 空值
- 默认值
- 合法值
- 非法值
- 格式校验
- 长度边界
- 数值边界
- 特殊字符
- 重复值
- 联动字段
- 类型切换后字段残留

#### 应测试内容
- 必填缺失时是否拦截
- 非法格式是否报错
- 边界值是否处理正确
- 错误提示是否准确
- 修正后是否可成功提交

#### 可写无需的条件
- 无需测试表单校验；原因：该界面无任何用户输入控件。

---

### F. 列表、表格、卡片数据展示

#### 必查内容
若界面存在集合型展示，必须检查：
- 列头 / 字段名
- 行数据 / 卡片数据
- 排序
- 截断
- 长文本显示
- 空值显示
- 单条与多条情况
- 更新后是否立即反映

#### 应测试内容
- 数据展示正确
- 新增/编辑/删除后列表同步更新
- 空数据与非空数据表现正确
- 特殊值、长文本、缺失值表现正确

#### 可写无需的条件
- 无需测试列表展示；原因：该界面不展示集合数据，仅包含单条详情内容。

---

### G. 搜索、筛选、排序、分页

#### 必查内容
若界面存在相关控件，必须检查：
- 搜索输入
- 筛选项
- 排序切换
- 分页切换
- 多条件组合
- 清空条件
- 无结果状态

#### 应测试内容
- 搜索命中
- 搜索无结果
- 筛选准确
- 排序正确
- 翻页正确
- 条件切换后数据正确
- 刷新或返回后状态是否符合预期

#### 可写无需的条件
- 无需测试搜索；原因：该界面无搜索输入或搜索能力。
- 无需测试分页；原因：该界面无分页控件，数据不分批展示。
- 无需测试排序；原因：界面不支持排序交互。

---

### H. 弹窗、抽屉、二次确认

#### 必查内容
若界面存在浮层类交互，必须检查：
- 是否能打开
- 是否能关闭
- 标题是否正确
- 默认值是否正确
- 提交是否成功
- 取消是否生效
- 遮罩点击行为
- 二次确认文案
- 浮层内滚动
- 浮层按钮是否可点击

#### 应测试内容
- 打开流程
- 关闭流程
- 提交流程
- 取消流程
- 误触/遮罩交互
- 移动端可操作性

#### 可写无需的条件
- 无需测试弹窗或抽屉；原因：该界面无浮层类交互。
- 无需测试二次确认；原因：该界面不存在危险操作或确认弹窗。

---

### I. 操作反馈与消息提示

#### 必查内容
- 成功提示
- 失败提示
- 警告提示
- 加载提示
- 空状态提示
- 表单错误提示
- 按钮禁用或处理中提示

#### 应测试内容
- 操作成功后提示正确
- 操作失败后提示准确
- 提示是否与真实结果一致
- 处理中是否防重复提交

#### 可写无需的条件
- 无需测试 toast；原因：该界面所有反馈均采用页面内固定提示区，无 toast 机制。
- 无需测试加载提示；原因：该界面无异步加载与长耗时操作。

---

### J. 状态变化

#### 必查内容
界面是否存在以下状态：
- 初始状态
- 加载中
- 成功态
- 失败态
- 空状态
- 禁用态
- 运行中 / 处理中
- 已完成 / 已提交 / 已生效

#### 应测试内容
- 状态切换是否正确
- 状态对应的文案/按钮/颜色/可操作性是否正确
- 状态切换后是否影响其他区域

#### 可写无需的条件
- 无需测试运行中状态；原因：该界面无异步执行类动作。
- 无需测试空状态；原因：该界面为固定信息页，不依赖数据集合。

---

### K. 异常场景

#### 必查内容
- 接口失败
- 权限不足
- 非法输入
- 重复提交
- 数据不存在
- 数据已删除
- 部分数据缺失
- 历史脏数据
- 非预期返回值

#### 应测试内容
- 失败时页面是否稳定
- 是否给出明确提示
- 是否出现错误残留状态
- 是否允许用户恢复操作

#### 可写无需的条件
原则上**不建议整项写无需**。  
若确实无外部依赖、无数据写入、无异步行为，可写：
- 无需测试接口失败回退；原因：该界面为纯静态页面，无接口请求与动态数据加载。

---

### L. 边界条件

#### 必查内容
- 0 条数据
- 1 条数据
- 多条数据
- 最大长度
- 最小长度
- 上下限值
- 特殊字符
- 长文本
- 历史格式数据
- 最后一条数据
- 最后一个关键角色 / 关键资源

#### 应测试内容
- 边界情况下是否仍能正常显示与操作
- 是否有保护逻辑
- 是否出现字段残留、状态错乱、列表错位等问题

#### 可写无需的条件
- 无需测试“最后一条删除”边界；原因：该界面不提供删除能力。
- 无需测试“最后一个管理员/关键资源”边界；原因：该界面不管理身份、角色或关键资源。

---

### M. 权限与身份

#### 必查内容
至少判断是否存在以下身份差异：
- 未登录用户
- 普通用户
- 管理员
- 当前用户本人
- 只读角色
- 特殊系统账号

#### 应测试内容
- 页面是否可访问
- 按钮是否可见
- 操作是否允许
- 越权是否被拦截
- 拦截后的跳转与提示是否正确

#### 可写无需的条件
- 无需测试多角色权限；原因：该产品无登录体系、无身份区分。

如果产品有登录体系，则不得整项写无需，至少要确认未登录与已登录差异是否存在。

---

### N. 数据一致性与持久化

#### 必查内容
- 操作后列表是否更新
- 详情是否更新
- 再打开编辑时值是否正确
- 刷新页面后是否保持
- 再次进入页面后是否保持
- 前端显示与实际结果是否一致

#### 应测试内容
- 成功提示后，验证实际数据变化
- 非成功提示时，验证数据未误写
- 界面与后端状态保持一致

#### 可写无需的条件
- 无需测试持久化；原因：该界面所有交互仅为本地临时态，不涉及数据保存。

---

### O. 页面跳转与返回链路

#### 必查内容
- 新建后跳转
- 保存后停留/返回
- 删除后返回
- 登录后回跳
- 取消后返回
- 面包屑 / 返回按钮

#### 应测试内容
- 跳转目标正确
- 返回后状态符合预期
- 回跳地址安全且正确

#### 可写无需的条件
- 无需测试页面跳转；原因：该界面为单页局部组件，不触发任何路由切换。

---

### P. 响应式与终端适配

#### 必查内容
至少考虑：
- 桌面端
- 移动端
- 窄屏
- 表格横向滚动
- 抽屉或侧边栏收起/展开
- 弹窗在小屏可操作性
- 按钮被遮挡或不可点

#### 应测试内容
- 关键主流程在桌面端可执行
- 关键主流程在移动端可执行
- 关键按钮在小屏下不被遮挡
- 重要信息在小屏下仍可见

#### 可写无需的条件
- 无需测试移动端；原因：该系统明确限定仅桌面端使用，且产品规范不支持移动端访问。

若产品未明确限制移动端，则不得轻易写无需。

### Q. 可恢复性与重复操作

#### 必查内容
- 保存失败后是否可重试
- 关闭弹窗后是否可重新进入
- 执行失败后是否可再次执行
- 重复点击是否会产生重复数据
- 刷新后是否可继续操作

#### 应测试内容
- 失败恢复
- 重试行为
- 防重复提交
- 重复执行稳定性

#### 可写无需的条件
- 无需测试重试；原因：该界面无提交、执行或保存动作。

---

### R. 可观察结果与验证证据

#### 必查内容
每个测试用例必须有明确验证点，不允许只写“点击成功”。

#### 验证证据可以包括
- 文案变化
- 按钮状态变化
- 列表数据变化
- URL 变化
- 弹窗出现/消失
- 分页信息变化
- 详情内容变化
- 刷新后数据保持
- 下载结果存在
- 日志内容出现

#### 如不适用
- 不允许整项写无需
- 原因：每条测试必须有可观察验证证据

## 五、标准输出结构（Agent 必须遵守）

Agent 针对每一个界面输出测试设计时，必须严格按以下结构：

### 0. 输入依据与范围说明
- 需求来源
- 文档来源
- 代码或现有页面来源
- 本次明确在范围内的页面
- 本次明确不在范围内的页面

- 页面名称

- 页面类型

- 说明原因

### 2. 页面名称

### 3. 页面目标
- 说明该页面是做什么的

### 4. 页面内功能拆分

- 页面主区域

- 建议归属的 spec 文件名

- 关键断言落点：UI 提示、URL、列表/详情、刷新后持久化、后端文件/接口/下载结果

若当前任务已知不是 Playwright，可省略本节；若当前任务已知使用 Playwright，则不得省略。

## 六、`无需` 的标准写法

Agent 在任何维度上写“无需”或“待确认”时，必须使用以下格式：

- `无需测试 <维度/能力>；原因：<明确、具体、可验证的原因>`
- `待确认 <维度/能力>；原因：<明确说明当前缺失了什么资料或证据>`

### 合格示例
- 无需测试分页；原因：该页面无分页控件，全部数据一次性展示。
- 无需测试删除流程；原因：该界面仅提供查看与编辑，不提供删除能力。
- 无需测试多角色权限；原因：该产品无登录与角色系统，所有用户权限一致。

### 不合格示例
- 无需分页
- 不需要测
- 这个没有
- 暂时不测
- 先不管

### `待确认` 合格示例
- 待确认多角色权限；原因：当前仅拿到管理员界面，缺少普通用户或只读角色说明。
- 待确认移动端适配；原因：资料未说明该页面是否要求支持移动端，现有文档也无对应产品约束。

## 七、测试用例最小完备标准

1. 页面目标已定义
2. 入口与访问已判断
3. 基础元素已判断
4. 主流程已覆盖
5. 表单/输入校验已判断
6. 数据展示已判断
7. 搜索/筛选/分页已判断
8. 弹窗/确认交互已判断
9. 提示反馈已判断
10. 状态变化已判断
11. 异常场景已判断
12. 边界条件已判断
13. 权限与身份已判断
14. 数据一致性已判断
15. 响应式已判断
16. 可恢复性已判断
17. 每个不适用项都已写 `无需 + 原因`
18. 每个不确定项都已写 `待确认 + 原因`
19. 已输出页面清单与页面边界
20. 已输出页面内功能拆分
21. 已判断运行项目：`chromium` / `mobile-chrome` / 两者都要

23. 已判断选择器与关键断言是否可稳定实现
24. 已判断是否需要唯一测试数据、防数据污染与清理回滚
25. 已判断是否需要真实下载、弹窗、接口响应、文件恢复等 Playwright 能力
26. 已判断是否需要客户端报错跟踪或失败证据
27. 已判断是否存在硬等待、顺序依赖、脆弱定位等稳定性风险

只要其中任何一项未判断，就不得声称“该界面测试点已完整覆盖”。

## 八、推荐的测试点模板

Agent 可以按以下模板产出单条测试点：

### 模板
- `编号`：TC-XXX
- `所属页面`：页面名称
- `所属功能区块`：列表 / 表单 / 弹窗 / 配置区 / 导航区 等

- `预期结果`：界面与数据应如何表现
- `验证证据`：通过什么观察点确认通过

### 示例
- 编号：TC-001

- 预期结果：创建成功提示出现，列表出现新记录，刷新后记录仍存在  
- 验证证据：提示文案、列表新行、刷新后仍可见  
- 所属维度：主流程 / 数据一致性

## 九、Agent 执行指令

当接到“为某个界面写测试用例”任务时，Agent 必须执行以下动作：

1. 先识别输入依据、资料来源与任务范围
2. 先输出页面清单，并标明 `在范围内 / 不在范围内 / 待确认`
3. 对范围内页面逐个拆分功能区块与交互能力
4. 再识别界面类型与页面目标
5. 再按本规范的全部维度逐项检查
6. 对每个维度输出：`需要` / `无需 + 原因` / `待确认 + 原因`
7. 最后再生成测试用例清单
8. 不得跳过权限、异常、边界、状态、响应式中的任一类分析
9. 不得仅根据页面表面元素判断覆盖完整
10. 不得省略不适用项说明或不确定项说明
11. 若当前任务使用 Playwright，必须补充实现映射、运行项目、数据隔离与稳定性约束
12. 不得把“能写出测试点”误判为“能稳定落地为 Playwright 用例”
13. 不得把资料中未明确出现的能力直接写成既定事实

## 十一、Playwright 项目级补充规则（当前项目强制）

当前项目已明确使用 `@playwright/test`，并且从现有配置可确认以下事实：
- 默认测试目录为 `tests/e2e/full`
- 默认运行项目包含 `chromium` 与 `mobile-chrome`
- 默认 `workers: 1`
- CI 下 `retries: 1`
- 失败时保留 `trace`，失败时截图，视频默认关闭
- 默认 `baseURL` 为 `http://127.0.0.1:58080`

因此，Agent 在本项目输出测试设计或测试代码时，除通用界面测试维度外，还必须同时满足以下 Playwright 约束。

### 1. 必须先判断运行项目

每条测试点都必须明确属于以下哪类之一：
- 仅桌面端：`chromium`
- 仅移动端：`mobile-chrome`
- 桌面端与移动端都必须覆盖

判断原则：
- 涉及后台管理主流程，默认至少覆盖 `chromium`
- 涉及首页可用性、搜索面板、小屏布局、侧边栏收起展开、弹窗小屏可操作性时，必须判断是否需要 `mobile-chrome`
- 若页面明确仅桌面支持，才允许写：`无需测试 mobile-chrome；原因：该后台仅桌面端支持，产品规范不要求移动端可用`

### 2. 必须先判断是否是真实全链路测试

本项目默认是 `full` E2E，不是前端孤立组件测试。  
因此默认要求：
- 走真实页面入口或真实 URL
- 使用真实登录流程或真实权限态
- 验证真实数据落地，而不是只验证 toast
- 非必要不得通过 mock、stub、`page.route()` 篡改结果

只有在任务明确要求“只测前端展示”或“隔离外部依赖”时，才允许偏离真实全链路。

### 3. `workers: 1` 不等于允许顺序依赖

即使当前配置串行执行，也不得把前一个测试留下的数据当成后一个测试的前置条件。  
每条测试仍应满足：
- 可单独运行
- 可重复运行
- 不依赖其他 spec 先执行
- 不依赖上一次运行残留数据

## 十二、Playwright 用例设计补充维度（强制逐项检查）

以下维度在当前项目中同样属于强制检查项。  
如果适用，必须写清测试方式；如果不适用，必须写 `无需 + 原因`。

### S. 用例粒度与 spec 拆分

#### 必查内容
- 一个 `test()` 是否只覆盖一个完整且连贯的业务目标
- 是否把多个无关模块硬塞进同一个测试
- spec 文件名是否能体现模块与能力
- 测试标题是否直接说明“验证什么行为成立”

#### 应测试内容
- 同一测试内的步骤都服务于一个业务结果
- 同一 spec 内聚焦同一模块或同一类风险
- 标题直接体现结果，例如“管理员可创建并删除用户”，而不是“用户管理测试”

#### 可写无需的条件
- 不允许整项写无需
- 原因：只要落地到 Playwright，就必然涉及测试粒度与文件归属

---

### T. 选择器稳定性

#### 必查内容
- 是否优先使用语义化定位：`getByRole`、`getByLabel`、`getByPlaceholder`
- 是否可使用稳定属性：`#id`、`name`、稳定表单字段
- 文本定位是否限定在明确容器内
- 是否使用了脆弱的 `.nth()`、深层 CSS 链、全页模糊文本匹配

#### 应测试内容
- 关键交互控件使用稳定定位
- 删除、编辑、批量操作等行内按钮，应绑定到对应行或对应卡片容器
- 若只能使用文本，应结合父容器、`hasText`、唯一字段缩小范围

#### 约束要求
- 定位优先级建议为：
  1. `getByRole` / `getByLabel`
  2. 稳定 `id` / `name`
  3. 带容器约束的文本定位
  4. `nth()` 仅作最后手段
- 不得默认用 `body` 文本匹配代替关键控件定位

#### 可写无需的条件
- 无需测试语义化选择器；原因：该页面缺少可用语义属性，且现有代码仅暴露稳定 `id/name`，已可满足稳定定位。

---

### U. 等待机制与异步同步点

#### 必查内容
- 点击、提交、跳转、下载、弹窗、接口返回后，真正的完成信号是什么
- 是否需要等待 URL 变化、元素状态变化、响应返回、下载事件、dialog 事件
- 是否存在动画、延迟渲染、异步刷新、多次请求串联

#### 应测试内容
- 使用 `expect(...)` 的自动等待能力确认最终状态
- 下载类操作应等待 `download`
- 弹窗确认类操作应监听 `dialog`
- 页面跳转应等待 URL 或稳定元素出现
- 数据刷新应等待具体行、字段、标签、计数变化

#### 约束要求
- 默认禁止把 `waitForTimeout` 当作主要同步手段
- 仅当页面存在无法消除的动画或浏览器行为延迟，且无法通过更稳定信号判断时，才允许少量 `waitForTimeout`
- 如使用固定等待，必须说明原因

#### 可写无需的条件
- 无需测试异步同步点；原因：该页面完全无提交、无跳转、无异步加载，所有断言均为同步静态展示。

---

### V. 断言层次与真实结果验证

#### 必查内容
- 是否同时验证了“界面反馈”与“真实结果”
- 写操作后是否验证刷新后仍成立
- 权限拒绝后是否验证数据未被误改
- 异常后是否验证页面未进入错误残留态

#### 应测试内容
对于创建、编辑、删除、导入、恢复、保存、执行类操作，至少应覆盖：
1. 操作触发成功或失败的即时反馈
2. 当前页面的结果变化
3. 刷新后或重新进入后的持久化结果
4. 必要时的后端或下载结果

#### 约束要求
- 只断言“出现成功提示”不算通过
- 只断言“出现失败提示”也不算完整，仍需确认数据未误写
- 若页面已有更精确的断言目标，不应长期停留在 `page.locator('body')`

#### 可写无需的条件
- 无需测试持久化结果；原因：该操作仅影响当前页临时展开/收起状态，不会写入任何配置或数据。

---

### W. 测试数据唯一性、隔离与清理

#### 必查内容
- 是否需要创建唯一 ID、唯一名称、唯一 slug、唯一域名
- 是否会修改全局配置、文件、锁定状态、Cookie 策略、备份数据、系统任务
- 操作后是否需要删除、回滚、恢复原文件
- 是否额外创建了 `browser.newContext()` 但未关闭

#### 应测试内容
- 创建型数据默认使用时间戳或等价唯一值，避免冲突
- 修改全局配置前先保存原值，测试结束后恢复
- 修改文件、锁定文件、备份文件等全局资源时，使用 `try/finally` 回滚
- 临时下载文件、导入文件优先写到测试输出目录
- 新建的额外浏览器上下文必须关闭

#### 约束要求
- 不得依赖固定名字反复创建同一条记录
- 不得污染后续测试环境
- 不得因为“本地一次能过”就省略恢复逻辑

#### 可写无需的条件
- 无需测试数据清理；原因：该用例只读访问，不创建、不修改、不删除任何数据，也不改变任何全局状态。

## 十三、当前项目推荐的 Playwright 实现约定

以下约定来自当前仓库已有测试写法；新增测试时应优先复用，而不是重新发明一套风格。

### 1. 登录、导航、弹窗与客户端报错跟踪

优先复用已有 helper：
- `loginAsDevAdmin(page)`
- `logout(page)`
- `clickAdminNav(page, name)`
- `submitVisibleModal(page)`
- `attachClientErrorTracking(page, options)`
- `ensureAdminSidebarOpen(page)`

适用原则：
- 管理后台主流程，默认优先使用 `loginAsDevAdmin(page)`
- 移动端后台导航，优先使用 `ensureAdminSidebarOpen(page)` 或 `clickAdminNav(...)`
- 模态框提交，优先使用 `submitVisibleModal(page)` 统一行为
- 高风险页面或复杂流程，优先挂载 `attachClientErrorTracking(page)`，并仅对白名单错误做定向豁免

### 2. 多上下文场景

以下场景应优先考虑 `browser.newContext()`：
- 不同登录身份并行对比
- 不同 IP、不同 Cookie、不同 `storageState`
- 退出登录后旧上下文失效验证
- remember me、session、权限隔离、未登录/已登录并行校验

### 3. 移动端场景

移动端测试建议遵循：
- 运行在 `mobile-chrome`
- 使用 `isMobile` 判断并在不适用项目下 `test.skip(...)`
- 重点验证小屏可点、侧边栏可展开、弹窗与搜索面板可操作、关键内容未被遮挡

### 4. 文件与下载

涉及导出、下载、上传、恢复类流程时：
- 下载应验证文件名、文件存在或内容结构
- 临时文件应放到 Playwright 输出目录
- 上传文件应使用明确的测试文件或运行时生成文件

## 十四、当前项目明确禁止的 Playwright 写法

- 禁止把成功标准写成“点击后没有报错”
- 禁止只断言 toast，而不验证真实数据结果
- 禁止大量依赖 `waitForTimeout`
- 禁止无约束地使用 `.nth()` 选择器
- 禁止让测试依赖前一个 spec 先执行
- 禁止创建会污染环境的全局数据而不恢复
- 禁止把 CI 的 `retries: 1` 当成修复不稳定测试的手段
- 禁止为了让测试通过而绕开真实权限、真实跳转、真实持久化

## 十五、面向当前项目的补充输出模板

当 Agent 为本项目某个页面输出测试设计时，建议在“测试点清单”后追加如下信息：

### 1. 建议 spec 归属
- 例如：`tests/e2e/full/settings-persistence.spec.ts`

### 2. 建议测试标题
- 例如：`settings persist after refresh and re-login`

### 3. 建议运行项目
- `chromium`
- `mobile-chrome`
- 或两者都要

### 4. 建议 helper / fixture
- 例如：`loginAsDevAdmin(page)`、`attachClientErrorTracking(page)`

### 5. 建议唯一数据
- 例如：`const ts = Date.now()`
- 例如：用户名、分组 ID、站点 slug、域名必须拼接时间戳

### 6. 建议清理 / 回滚
- 例如：恢复被修改的设置值
- 例如：`finally` 恢复文件内容
- 例如：关闭额外 `browser context`

### 7. 关键断言
- 例如：成功提示 + 列表行变化 + 刷新后仍存在
- 例如：失败提示 + 数据未落地 + URL 未越权进入

## 十六、页面边界、功能归属与防越界判定规则

本节用于确保 Agent 在写测试示例时，先把“归属边界”判清，再写测试点。

### 1. 什么算一个页面

优先按以下信号判断页面：
- 独立 URL 或路由
- 独立菜单入口
- 独立标题与业务目标
- 独立数据上下文
- 独立保存/提交/执行入口

满足其中任意两项以上，通常应视为独立页面。

### 2. 什么算页面内功能区块

若同一页面内部存在明显独立的区域，应拆成不同功能区块，例如：
- 列表区
- 筛选区
- 表单区
- 配置区
- 日志区
- 统计区
- 弹窗区
- 批量操作区

每个功能区块都应单独识别：
- 核心能力
- 输入项
- 输出结果
- 风险点

### 3. 什么算共享能力

以下能力默认优先视为共享能力，而不是某一个业务页面私有能力：
- 顶部导航
- 侧边栏导航
- 全局搜索
- 登录态校验
- 通用上传
- 通用下载
- 通用确认弹窗
- 通用提示组件

若共享能力在多个页面都出现，应按以下原则处理：
- 共享组件自身行为，集中在共享能力测试中覆盖
- 共享组件与具体业务数据的结合行为，仍需在业务页面补充验证

### 4. 什么情况不应并入当前页面测试

出现以下情况时，默认不应并入当前页面测试：
- 真实归属于其他菜单或其他 URL
- 需要进入另一个完整页面才能完成
- 业务目标已发生切换
- 数据上下文已变化为另一类实体
- 测试重点已经从“当前页面行为”转移到“目标页面行为”

此时必须拆分写明：
- 当前页面负责验证什么
- 目标页面负责验证什么

### 5. 跨页面流程的写法要求

跨页面流程可以写，但必须按页面分层描述：
- 当前页面的入口与跳转动作
- 目标页面的到达验证
- 目标页面的核心行为
- 返回原页面后的结果验证

禁止把整条链路写成无归属的连续步骤，而不标注每个步骤属于哪个页面。

### 6. 对 Agent 的硬约束

Agent 在任何情况下都不得：
- 把共享能力误写成某业务页面独有能力
- 把另一个页面的行为误并入当前页面
- 因为页面上“看起来有入口”就把整个下游页面都纳入本次范围
- 因为文档没写就自行删除某个测试维度
- 因为资料不足就擅自补完业务规则

### 7. 推荐的页面覆盖矩阵字段

若任务涉及多个页面，推荐先输出如下矩阵字段：
- 页面名称
- 页面入口
- 页面目标
- 页面内功能区块
- 核心能力
- 风险等级
- 是否在本次范围内
- 不在范围内或待确认的原因

### 8. 推荐的测试点追溯字段

每条测试点最好能追溯到以下信息：
- 来源页面
- 来源功能区块
- 来源能力
- 对应测试维度
- 来源依据：需求 / 文档 / 代码 / 现有实现

只要无法追溯到页面与能力归属，该测试点就不应视为合格。
