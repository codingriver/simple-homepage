# 文件系统模块需求与设计文档

版本：v1.0  
状态：待开发  
适用范围：后台文件管理、本机文件管理、远程主机文件管理、后续与 SSH / Host-Agent 联动扩展

---

## 1. 目标

文件系统模块的目标不是简单做一个“能看目录”的页面，而是做成这个后台里的统一文件工作台，覆盖：

- 宿主机本机文件管理
- 远程主机文件管理
- 浏览器内基础编辑
- 上传、下载、压缩、解压
- 权限与属主属组管理
- 风险操作审计
- 后续与 Web 终端、计划任务、SSH 密钥、备份模块联动

最终体验目标参考 1Panel / 宝塔的文件管理，但当前阶段不追求一次做到全部复杂能力，优先做稳定、可控、可审计、适合容器环境的版本。

---

## 2. 产品定位

文件系统模块定位为后台核心基础能力，服务以下场景：

- 在后台直接编辑配置文件，如 `nginx.conf`、`docker-compose.yml`、`sshd_config`
- 管理计划任务脚本、日志、输出文件
- 浏览和维护站点挂载目录、备份目录、证书目录
- 通过远程 SSH 管理其它 VPS / 服务器文件
- 为后续“上传部署包、解压发布、模板下发、日志排障”提供基础设施

这个模块应优先保证：

- 可控：默认只开放允许的路径与主机
- 稳定：不能因为大文件、二进制文件、失败请求把页面拖死
- 可审计：关键操作要记录
- 一致：本机与远程主机操作体验尽量统一

## 4. 使用对象与权限模型

### 4.1 用户角色

- `admin`
  - 完整文件系统权限
- `host_admin`
  - 主机与文件管理权限
- `host_viewer`
  - 只读文件浏览权限

### 4.2 建议细粒度权限

建议新增或明确以下权限点：

- `fs.view`
- `fs.read`
- `fs.write`
- `fs.delete`
- `fs.upload`
- `fs.download`
- `fs.archive`
- `fs.permission`
- `fs.remote`
- `fs.audit`

### 4.3 权限控制原则

- 没有 `fs.write` 时：
  - 不显示新建、保存、上传、重命名、移动、复制、删除按钮
- 没有 `fs.permission` 时：
  - 不显示 `chmod/chown/chgrp`
- 没有 `fs.archive` 时：
  - 不显示压缩/解压
- 没有 `fs.remote` 时：
  - 只能看本机，不显示远程主机切换

## 5. 路径与安全边界

这是文件系统模块最关键的设计点。

### 5.1 本机路径策略

本机不能默认无限制暴露整个容器文件系统，必须采用“白名单根目录”策略。

建议默认白名单：

- `/var/www/nav/data`
- `/var/www/nav/public`
- `/var/www/nav/admin`
- `/var/www/nav/shared`
- `/etc/nginx`
- `/etc/ssh`
- `/var/log`

后续在系统设置中允许管理员追加白名单路径。

### 5.2 容器与宿主机关系

当前项目运行在容器内，因此要明确：

- 默认文件管理看到的是容器视角文件系统
- 若某路径是宿主机挂载进容器的卷，则用户实际操作的是宿主机对应目录
- 若用户需要完整宿主机文件管理，应通过 `host-agent` 扩展宿主机文件能力，而不是直接假设容器能看到宿主机全盘

### 5.3 路径校验要求

- 所有本地路径必须：
  - 归一化
  - 禁止 `..` 逃逸
  - 禁止软链绕过白名单根
- 所有远程路径必须：
  - 做 shell 转义
  - 不允许原样拼接用户输入到命令

## 7. 核心交互设计

### 7.1 文件打开逻辑

- 小文本文件：
  - 直接读取并展示到编辑器
- 二进制文件：
  - 不直接编辑，只提供下载、属性、十六进制占位提示
- 大文件：
  - 超过阈值后不自动全文加载
  - 提示用户下载或仅查看前 N KB

建议阈值：

- 文本在线编辑阈值：2 MB
- 预览阈值：512 KB

### 7.2 删除逻辑

- 删除文件直接确认
- 删除目录需二次确认，并明确是否递归
- 批量删除需显示条目数

### 7.3 上传逻辑

- 支持单文件上传
- 支持多文件上传
- 显示进度
- 上传冲突时支持：
  - 覆盖
  - 跳过
  - 自动重命名

### 7.4 复制 / 剪切 / 粘贴

采用前端暂存剪贴板状态：

- `copy`: 记录源路径列表
- `cut`: 记录源路径列表与操作类型
- `paste`: 调用后端执行

后端必须再次校验：

- 目标路径是否允许
- 是否会覆盖
- 是否跨设备

### 7.5 压缩 / 解压

- 压缩时支持选择输出文件名
- 解压时支持选择目标目录
- 检测压缩包格式并提示不支持类型

## 8. 本机与远程统一抽象

### 8.1 统一目标模型

建议继续沿用 SSH 模块已经形成的 target 设计：

- `local`
- `remote`

统一传输结构：

```json
{
  "target": {
    "type": "local"
  }
}
```

或：

```json
{
  "target": {
    "type": "remote",
    "host": "...",
    "port": 22,
    "username": "root",
    "auth_type": "key"
  }
}
```

### 8.2 后端抽象层

建议文件能力拆分为：

- `admin/shared/file_manager_lib.php`
  - 页面层调用封装
- `cli/host_agent.php`
  - 本机 / 远程实际执行层
- `admin/file_api.php`
  - AJAX API 层

不要再把完整文件 API 继续堆进 `host_api.php`，否则 SSH 与文件系统边界会越来越混乱。

- `scandir`

- `is_dir`
- `filesize`
- `filemtime`
- `file_get_contents`
- `file_put_contents`
- `mkdir`
- `rename`
- `unlink`
- `rmdir`
- `chmod`
- `chown`
- `chgrp`

目录递归删除、复制可封装为内部函数。

### 11.2 远程执行

继续基于 SSH 命令执行：

- `find`
- `stat`
- `tar`
- `unzip`
- `chmod`
- `chown`
- `chgrp`
- `mv`
- `cp`
- `rm`

- 直接 PHP 上传流写文件

远程：

- 小文件可走 base64 传输
- 大文件后续建议改为分片上传，v1 可先限制大小

## 12. 风险点

### 12.1 高风险操作

- 删除目录
- 批量删除
- 修改权限
- 修改属主
- 覆盖已有文件
- 解压未知压缩包

这些操作必须：

- 二次确认
- 记录审计
- 失败时返回明确错误

### 12.2 性能风险

- 大目录遍历卡顿
- 大文件读取卡顿
- 远程递归搜索耗时高
- 远程传输 base64 内存占用高

对应策略：

- 大目录分页
- 大文件阈值限制
- 搜索限深度或限结果数
- 上传大小限制

### 12.3 安全风险

- 路径穿越
- 软链逃逸
- 远程命令注入
- 覆盖关键系统文件

对应策略：

- 统一路径规范化
- 白名单根目录限制
- shell 参数全部转义
- 关键路径增加风险提示

## 15. 推荐开发顺序

建议先做：

1. 独立 `files.php` 页面骨架
2. 本机文件白名单与路径安全
3. 本机文件浏览与文本编辑
4. 上传下载
5. 删除 / 重命名 / 新建
6. 权限操作
7. 压缩解压
8. 收藏 / 最近访问
9. 远程主机统一接入

原因：

- 先把本机模型和页面结构做稳
- 再把远程能力接进来
- 否则一开始就混着做，本机和远程两套逻辑会缠在一起

## 16. 与现有模块的关系

- 与 SSH 模块关系：
  - 复用目标主机模型、连接能力、审计能力
- 与计划任务关系：
  - 可直接管理 `data/tasks` 下脚本与日志
- 与备份模块关系：
  - 后续可直接浏览备份目录并下载
- 与通知中心关系：
  - 后续可加入高风险文件操作通知

---

## 17. 结论

文件系统模块值得做，而且应当作为后台下一阶段核心功能。

推荐策略不是继续在 `hosts.php` 里补按钮，而是单独建设：

- `files.php`
- `file_api.php`
- `file_manager_lib.php`

先做本机安全白名单文件管理，再扩到远程主机，逐步接近 1Panel / 宝塔的文件系统体验。
