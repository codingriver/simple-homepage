# RiverOps 项目测试规划

本文档记录当前 RiverOps 运维控制台的测试边界与发布基线。

## 1. 测试目标

- 防止认证、Cookie、会话和权限回归。
- 验证 DNS、DDNS、域名有效期和计划任务核心流程。
- 验证运行环境安装任务的状态收敛与日志轮询。
- 验证备份、Webhook、日志和系统设置的数据一致性。
- 验证 Docker 在 amd64/arm64、低内存和空闲场景下的基础可用性。

## 2. 测试分层

| 层级 | 工具 | 重点 |
|---|---|---|
| 单元测试 | PHPUnit 11 | 认证、锁、配置、业务共享库、CLI 和 Docker 配置 |
| 桌面 E2E | Playwright Chromium | 完整后台操作流程 |
| 移动端 E2E | Playwright mobile-chrome | Pixel 7 布局、交互与表单可达性 |
| 性能 | Lighthouse CI | Performance、Accessibility、Best Practices |
| 容器验证 | Docker Compose | 构建、启动、运行用户、配置文件和路径 |

## 3. 默认命令

```bash
vendor/bin/phpunit
npm exec tsc -- --noEmit
npm run test:e2e:full:chromium
npm run test:e2e:full:mobile-chrome
npm run test:perf
docker compose config --quiet
```

Docker 测试环境：

```bash
docker compose -f local/docker-compose.yml -f local/docker-compose.dev.yml -f local/docker-compose.test.yml run --rm playwright-full
docker compose -f local/docker-compose.yml -f local/docker-compose.dev.yml -f local/docker-compose.test.yml run --rm playwright-mobile
```

## 4. 功能覆盖

| 模块 | 最低覆盖要求 |
|---|---|
| 安装与登录 | 首装、CSRF、错误密码、IP 锁定、记住我、完成页 Cookie |
| 会话 | 多设备上限、多选踢下线、撤销、并发 touch |
| 权限 | admin/user、权限点拒绝、AJAX 401 |
| DNS/DDNS | CRUD、执行日志、失败信息、数据一致性 |
| 域名有效期 | RDAP 缓存、刷新、删除和紧急状态 |
| 计划任务 | 多运行类型、脚本保存、依赖安装、超时、锁和日志分页 |
| 运行环境 | 检测、安装任务、进度、重新进入恢复、异常进程收敛 |
| 备份 | 创建、下载、删除、恢复和非法结构拒绝 |
| 通知 | Webhook 配置、测试发送、失败展示 |
| 日志与调试 | 分类、筛选、分页、清理和构建信息 |
| 设置 | Cookie 策略、`panel_domain`、访问日志开关和自定义 CSS |

## 5. 重命名专项验收

破坏性重命名完成后必须验证：

1. 源码不存在旧项目名、旧环境变量、旧 Cookie、旧运行用户或旧容器路径。
2. 容器内项目目录为 `/var/www/riverops`。
3. Web、PHP-FPM 和 Nginx 进程用户为 `riverops`。
4. Cookie 名为 `riverops_session` 和 `riverops_php_session`。
5. 环境变量只接受 `RIVEROPS_*`。
6. Nginx、PHP-FPM、PHP、sudoers 和日志文件使用 `riverops` 文件名。
7. 使用空数据目录可以完成首次安装、登录和后台访问。
8. 旧配置字段和旧备份不作为支持场景。

## 6. CI 基线

- Pull Request：PHPUnit、TypeScript、Compose 解析和关键 Playwright 回归。
- 默认分支：桌面端、移动端和 Lighthouse。
- 标签发布：完整测试通过后构建 `linux/amd64`、`linux/arm64` 镜像。
- 任一认证、权限、锁或数据一致性测试失败时禁止发布。

## 7. 维护规则

- 新增页面必须同时新增权限与 CSRF 回归。
- 新增 AJAX action 必须覆盖成功、鉴权失败、参数错误和未知 action。
- 修改容器路径或环境变量时，必须同步更新 Docker、Compose、文档和测试。
- 修复并发问题时必须增加可重复执行的锁或一致性测试。
- E2E 用例不得依赖执行顺序，测试产生的数据必须在结束后清理。
